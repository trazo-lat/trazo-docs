The `heyllave db` command group manages the local PostgreSQL Docker container used during development and provides utilities for inspecting and interacting with the database.

## `heyllave db local`

Start a local PostgreSQL instance inside Docker. On the first run the container image is pulled and the container **`heyllave-local-db`** is created. Subsequent runs start the existing container.

```bash
heyllave db local
```

**Output:**

```
[heyllave] Starting container heyllave-local-db...
[heyllave] PostgreSQL ready on localhost:5432
```

### Flags

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--stop` | bool | `false` | Stop the running `heyllave-local-db` container instead of starting it |
| `--port` | int | `5432` | Map the container's PostgreSQL port to this host port |

```bash
# Stop the local database container
heyllave db local --stop

# Start on a non-default port
heyllave db local --port 5433
```

<Aside type="caution">
Changing `--port` only affects the host-side port mapping. Update `database.url` in your `heyllave.yaml` (or set `HEYLLAVE_DATABASE_URL`) to match the new port so the server and CLI can connect.
</Aside>

## `heyllave db ping`

Test that the CLI can reach the configured database. Useful for validating connection strings in CI or after changing configuration.

```bash
heyllave db ping
```

**Success output:**

```
[heyllave] Connected to database at localhost:5432 (latency: 2ms)
```

**Failure output:**

```
[heyllave] ERROR: could not connect to database: dial tcp 127.0.0.1:5432: connect: connection refused
```

The command exits with a non-zero code on failure, making it safe to use in shell scripts and CI pipelines:

```bash
heyllave db ping && echo "Database is reachable"
```

## `heyllave db status`

Display a summary of the current database: connection info, schema list, and the tables present in each schema.

```bash
heyllave db status
```

**Example output:**

## `heyllave db connect`

Open an interactive `psql` session connected to the configured database. The session inherits all `psql` features including tab completion and `\dt`, `\d`, and `\copy` meta-commands.

```bash
heyllave db connect
```

The CLI resolves the connection string from `heyllave.yaml` (or environment variables) and passes it directly to `psql`. Ensure `psql` is installed and on your `PATH`.

## `heyllave db switch <tenant>`

Switch the CLI's active schema context to the specified tenant's schema. Once switched, subsequent `heyllave db connect` sessions and other schema-aware commands will target that tenant's schema by default.

```bash
heyllave db switch acme
```

The `<tenant>` argument accepts either the tenant's **subdomain** or its **schema name** (e.g., `tenant_acme`).

**Example:**

```bash
# Switch to tenant "acme" and open a psql session scoped to its schema
heyllave db switch acme
heyllave db connect
# Inside psql:
# SET search_path = tenant_acme;  (already set by the CLI)
# \dt
```

<Aside type="tip">
Use `heyllave db status` to list all available tenant schemas before switching.
</Aside>
