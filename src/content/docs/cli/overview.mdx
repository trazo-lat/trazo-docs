The `heyllave` CLI is the primary developer tool for managing every aspect of a Heyllave project from the terminal. It covers local database lifecycle, schema migrations, and module development utilities — all driven by the same `heyllave.yaml` configuration file used by the server.

## Installation

The CLI is built from the `heyllave-cli` repository:

```bash
git clone https://github.com/your-org/heyllave-cli
cd heyllave-cli
go build -o heyllave ./cmd/heyllave
sudo mv heyllave /usr/local/bin/
```

Verify the installation:

```bash
heyllave --version
```

## Global Flags

These flags are accepted by every `heyllave` command and must be placed before the subcommand name.

| Flag | Short | Type | Default | Description |
|------|-------|------|---------|-------------|
| `--config` | `-c` | string | `heyllave.yaml` | Path to the Heyllave configuration file |
| `--verbose` | `-v` | bool | `false` | Enable verbose output including debug logs |

```bash
# Use a custom config file
heyllave --config /etc/heyllave/staging.yaml db status

# Enable verbose output
heyllave --verbose migrate up
```

## Configuration Precedence

When the CLI resolves a configuration value it evaluates sources in the following order — higher entries win over lower ones:

| Priority | Source | Example |
|----------|--------|---------|
| 1 (highest) | CLI flags | `--config ./my.yaml` |
| 2 | Environment variables | `HEYLLAVE_DATABASE_URL=postgres://...` |
| 3 | Project config file | `heyllave.yaml` in the current directory |
| 4 | User-local config | `~/.heyllave/local.json` |
| 5 (lowest) | Built-in defaults | port `5432`, env `local` |

<Aside type="tip">
Use `~/.heyllave/local.json` for secrets and personal overrides (database passwords, JWT secrets) so they are never committed to source control. The file supports the same keys as `heyllave.yaml` in JSON form.
</Aside>

## Command Groups

The CLI is organized into three top-level command groups:

| Group | Description |
|-------|-------------|
| `heyllave db` | Manage the local PostgreSQL Docker container and inspect database state |
| `heyllave migrate` | Run, rollback, and inspect Atlas-format schema migrations |
| `heyllave module` | Validate and generate code from `module.yaml` manifests |

## Local Development Container

For local development the CLI manages a Docker container named **`heyllave-local-db`** running PostgreSQL. This container is created and started by `heyllave db local` and can be stopped with `heyllave db local --stop`.

```bash
# Start the local database (creates the container on first run)
heyllave db local

# Stop the running container
heyllave db local --stop
```

The container name `heyllave-local-db` is fixed and used by the other `db` subcommands to resolve the connection string when no explicit `database.url` is set in the config file.

## Getting Help

Every command and subcommand supports `--help`:

```bash
heyllave --help
heyllave db --help
heyllave migrate --help
heyllave module --help
```
