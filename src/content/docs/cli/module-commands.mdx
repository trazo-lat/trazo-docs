The `trazo module` command group provides tooling for working with Trazo module manifests. Modules are declared in a `module.yaml` file and the CLI can validate that manifest against the full specification and generate boilerplate Go code from it.

## `trazo module validate [path]`

Validate a `module.yaml` file against the Trazo module specification. The command checks the schema structure, required fields, permission key format, scope filter syntax, and cross-field consistency rules.

```bash
# Validate the module.yaml in the current directory
trazo module validate

# Validate a specific file or directory
trazo module validate ./modules/repairs/module.yaml
trazo module validate ./modules/repairs/
```

### What is Validated

| Category | Checks |
|----------|--------|
| Schema | All top-level keys are recognized; no unknown fields present |
| Required fields | `name`, `version`, `type`, and `description` are present and non-empty |
| Permission format | Permission keys follow the `<module>.<action>` pattern (e.g., `repairs.create`) |
| Scope filters | Filter expressions use supported placeholders (`:user_id`, `:user_team_ids`) |
| Transitions | Referenced statuses exist in the `statuses` section; `from`/`to` values are valid |
| Dependencies | All declared `dependencies` match known module names |
| Event names | `emits` and `subscribes` event names follow the `<module>.<event>` convention |

### Output

The command prints warnings and errors separately:

```
[trazo] Validating modules/repairs/module.yaml

  WARN  permissions[2]: description is empty — add a human-readable description
  ERROR permissions[3]: key "repairs.Delete" must be lowercase (got "repairs.Delete")
  ERROR transitions[1].to: status "closed" is not declared in statuses

Validation failed: 2 error(s), 1 warning(s)
```

A clean file produces:

```
[trazo] Validating modules/repairs/module.yaml
[trazo] OK — no errors or warnings
```

The command exits with code `0` only when there are zero errors. Warnings do not affect the exit code.

<Aside type="tip">
Run `trazo module validate` in a CI step before merging changes to a `module.yaml`. Treating warnings as errors can be enforced by checking the warning count in the output.
</Aside>

## `trazo module generate`

Read a validated `module.yaml` and emit Go source files into the module's package directory. The generator produces constants, configuration types, and error code definitions so you never manually keep Go code in sync with the manifest.

```bash
# Generate from module.yaml in the current directory
trazo module generate

# Generate from a specific file
trazo module generate --module ./modules/repairs/module.yaml
```

### Generated Files

| File | Contents |
|------|----------|
| `zz_permissions.go` | Typed permission key constants for every entry in `permissions` |
| `zz_statuses.go` | String constants for every status declared in `statuses` |
| `zz_errors.go` | Error code constants and factory helpers using the module's error prefix |
| `zz_config.go` | A typed `Config` struct matching the `settings` section of the manifest |

All generated files carry a `// Code generated by trazo module generate. DO NOT EDIT.` header and are prefixed with `zz_` so they are easy to identify and exclude from manual review.

### Example: Generated Permissions

Given this `module.yaml` fragment:

```yaml
permissions:
  - key: repairs.create
    name: Create Repair
    scope: tenant
  - key: repairs.view
    name: View Repair
    scope: tenant
```

The generator produces `zz_permissions.go`:

```go
// Code generated by trazo module generate. DO NOT EDIT.

package repairs

const (
    PermCreate = "repairs.create"
    PermView   = "repairs.view"
)
```

### Internals

The generator is implemented in the `internal/generate/` package of the `trazo-cli` repository. It uses Go's `text/template` engine with the parsed `module.yaml` as data. Templates are embedded in the binary so no external template files are required.

<Aside type="caution">
Always run `trazo module validate` before `trazo module generate`. The generator assumes a valid manifest and may produce incorrect code if the manifest contains errors.
</Aside>
