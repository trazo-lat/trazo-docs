The `trazo migrate` command group manages database schema migrations using the [Atlas](https://atlasgo.io/) migration format. Migration files live in the `migrations/` directory of the `trazo-core-service` repository and are applied in sequence by their numeric prefix.

## Migration File Format

Trazo uses the Atlas SQL migration format. Each file contains an `-- +down` separator that divides the **up** (apply) and **down** (rollback) SQL statements:

```sql
-- 0003_create_repairs.sql

CREATE TABLE tenant_schema.repairs (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    vehicle_id  UUID NOT NULL,
    status      VARCHAR(50) NOT NULL DEFAULT 'pending',
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- +down

DROP TABLE IF EXISTS tenant_schema.repairs;
```

---

## `trazo migrate up`

Apply all pending migrations in ascending order.

```bash
trazo migrate up
```

**Example output:**

```
[trazo] Applying 0003_create_repairs.sql ... OK
[trazo] Applying 0004_add_repair_notes.sql ... OK
[trazo] All migrations applied. Database is up to date.
```

### Flags

| Flag | Type | Description |
|------|------|-------------|
| `--to <version>` | string | Migrate up only to the specified version number (inclusive). Does not apply later migrations. |

```bash
# Apply migrations only up to version 0005
trazo migrate up --to 0005
```

<Aside type="tip">
`--to` is useful when you need to match a specific version during staged rollouts or when testing individual migration pairs.
</Aside>

---

## `trazo migrate down`

Rollback the most recently applied migration using its `-- +down` section.

```bash
trazo migrate down
```

**Example output:**

```
[trazo] Rolling back 0004_add_repair_notes.sql ... OK
```

### Flags

| Flag | Type | Description |
|------|------|-------------|
| `--to <version>` | string | Rollback all migrations down to (but not including) the specified version. |

```bash
# Rollback everything after version 0002
trazo migrate down --to 0002
```

<Aside type="caution">
Rollbacks are irreversible in production unless you have a database backup. Always test migration pairs locally before deploying.
</Aside>

## `trazo migrate status`

Show which migrations have been applied and which are still pending.

```bash
trazo migrate status
```

**Example output:**

```
Version   File                          Status    Applied At
────────  ────────────────────────────  ────────  ──────────────────────
0001      0001_initial_schema.sql       applied   2025-11-01 09:12:03
0002      0002_add_roles.sql            applied   2025-11-05 14:33:21
0003      0003_create_repairs.sql       applied   2026-01-10 08:44:55
0004      0004_add_repair_notes.sql     pending   —
```

Exit code `0` when all migrations are applied; non-zero when pending migrations exist (useful in CI pre-flight checks).

---

## `trazo migrate history`

Show the full migration history including the exact timestamp each migration was applied.

```bash
trazo migrate history
```

**Example output:**

```
Version   File                          Applied At               Duration
────────  ────────────────────────────  ───────────────────────  ────────
0001      0001_initial_schema.sql       2025-11-01 09:12:03 UTC  142ms
0002      0002_add_roles.sql            2025-11-05 14:33:21 UTC  38ms
0003      0003_create_repairs.sql       2026-01-10 08:44:55 UTC  65ms
```

Unlike `migrate status`, `history` shows all previously applied migrations regardless of whether their files still exist in the `migrations/` directory.

## `trazo migrate create <name>`

Scaffold a new migration file in the `migrations/` directory. The file is prefixed with the next sequential version number and contains placeholder sections for the up and down SQL.

```bash
trazo migrate create add_repair_photos
```

**Creates:** `migrations/0005_add_repair_photos.sql`

**File contents:**

```sql
-- 0005_add_repair_photos.sql
-- Created: 2026-03-03T10:00:00Z

-- Write your migration SQL here


-- +down

-- Write your rollback SQL here
```

<Aside type="tip">
The `<name>` argument should be lowercase and use underscores. The CLI automatically prepends the next available version number and appends `.sql`.
</Aside>

## `trazo migrate diff`

Compare the current state of the database schema against the migration files and report any divergence. Useful for detecting manual schema changes or out-of-band modifications.

```bash
trazo migrate diff
```

**Example output (no differences):**

```
[trazo] Schema matches migration history. No differences found.
```

**Example output (differences detected):**

```
[trazo] WARNING: Schema divergence detected.
  Table "tenant_schema.repairs" has column "technician_notes" not present in any migration.
  Run `trazo migrate create` to capture this change.
```

`migrate diff` is read-only and never modifies the database or migration files.
