The `heyllave migrate` command group inspects and reconciles Heyllave's declarative, desired-state SQL migrations. Migration files live under `migrations/` in the `heyllave-core-service` repository, split into three tiers:

- `migrations/core/public/` — global cross-tenant tables, applied once.
- `migrations/core/tenant/` — per-tenant core subsystem tables, applied to every `tenant_<slug>` schema.
- `migrations/modules/<name>.sql` — module-owned per-tenant tables, applied to each tenant schema where the module is enabled.

See [Migrations](/guides/migrations/) for the full conceptual model.

## Migration File Format

Each file is plain desired-state SQL. There are no numeric prefixes and no `-- +down` separator — Heyllave uses `pg-schema-diff` to compute whatever DDL is needed to bring the live schema into agreement with the file.

```sql
-- migrations/modules/repairs.sql

CREATE TABLE IF NOT EXISTS repairs (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    vehicle_id  UUID NOT NULL,
    status      VARCHAR(50) NOT NULL DEFAULT 'pending',
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

Note: there is no `tenant_id` column. The table lives in `tenant_<slug>.repairs` once provisioned, and the schema name is the tenancy boundary.

## `heyllave migrate up`

Apply all pending migrations in ascending order.

```bash
heyllave migrate up
```

**Example output:**

```
[heyllave] Applying 0003_create_repairs.sql ... OK
[heyllave] Applying 0004_add_repair_notes.sql ... OK
[heyllave] All migrations applied. Database is up to date.
```

### Flags

| Flag | Type | Description |
|------|------|-------------|
| `--to <version>` | string | Migrate up only to the specified version number (inclusive). Does not apply later migrations. |

```bash
# Apply migrations only up to version 0005
heyllave migrate up --to 0005
```

<Aside type="tip">
`--to` is useful when you need to match a specific version during staged rollouts or when testing individual migration pairs.
</Aside>

## `heyllave migrate down`

Roll the schema back to a previously recorded state. Because Heyllave migrations are declarative, there is no per-file `-- +down` script; the engine instead diffs against the previous recorded snapshot in `migration_history` and emits the reverse DDL.

```bash
heyllave migrate down
```

**Example output:**

```
[heyllave] Rolling back 0004_add_repair_notes.sql ... OK
```

### Flags

| Flag | Type | Description |
|------|------|-------------|
| `--to <version>` | string | Rollback all migrations down to (but not including) the specified version. |

```bash
# Rollback everything after version 0002
heyllave migrate down --to 0002
```

<Aside type="caution">
Rollbacks are irreversible in production unless you have a database backup. Always test migration pairs locally before deploying.
</Aside>

## `heyllave migrate status`

Show which migrations have been applied and which are still pending.

```bash
heyllave migrate status
```

**Example output:**

```
Version   File                          Status    Applied At
────────  ────────────────────────────  ────────  ──────────────────────
0001      0001_initial_schema.sql       applied   2025-11-01 09:12:03
0002      0002_add_roles.sql            applied   2025-11-05 14:33:21
0003      0003_create_repairs.sql       applied   2026-01-10 08:44:55
0004      0004_add_repair_notes.sql     pending   —
```

Exit code `0` when all migrations are applied; non-zero when pending migrations exist (useful in CI pre-flight checks).

## `heyllave migrate history`

Show the full migration history including the exact timestamp each migration was applied.

```bash
heyllave migrate history
```

**Example output:**

```
Version   File                          Applied At               Duration
────────  ────────────────────────────  ───────────────────────  ────────
0001      0001_initial_schema.sql       2025-11-01 09:12:03 UTC  142ms
0002      0002_add_roles.sql            2025-11-05 14:33:21 UTC  38ms
0003      0003_create_repairs.sql       2026-01-10 08:44:55 UTC  65ms
```

Unlike `migrate status`, `history` shows all previously applied migrations regardless of whether their files still exist in the `migrations/` directory.

## `heyllave migrate create <name>`

Scaffold a new desired-state SQL file. The CLI places the file in the appropriate tier (`migrations/core/public/`, `migrations/core/tenant/`, or `migrations/modules/`) based on the flags you pass.

```bash
heyllave migrate create repairs --tier modules
```

**Creates:** `migrations/modules/repairs.sql`

**File contents:**

```sql
-- migrations/modules/repairs.sql
-- Desired state for the repairs module.
-- Edit this file to describe what the schema should look like.
```

<Aside type="tip">
The `<name>` argument should be lowercase and use underscores. There are no numeric prefixes and no `-- +down` separators — Heyllave migrations describe the desired state and the diff engine computes the DDL needed to reach it.
</Aside>

## `heyllave migrate diff`

Compare the current state of the database schema against the migration files and report any divergence. Useful for detecting manual schema changes or out-of-band modifications.

```bash
heyllave migrate diff
```

**Example output (no differences):**

```
[heyllave] Schema matches migration history. No differences found.
```

**Example output (differences detected):**

```
[heyllave] WARNING: Schema divergence detected in tenant_acme.
  Table "repairs" has column "technician_notes" not present in migrations/modules/repairs.sql.
  Update the SQL file to capture (or remove) this change.
```

`migrate diff` is read-only and never modifies the database or migration files.
