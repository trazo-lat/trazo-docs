Trazo supports multiple authentication methods — `password`, `oauth2`, `magic_link`, and `webauthn`. By default, every membership allows all methods. The auth method management API lets tenant admins **restrict which methods each member can use** and **configure tenant-wide defaults** for new memberships.

Requires the `members.manage_auth` permission for GET/PUT operations on the dedicated endpoints. Auth methods can also be updated via the general `PUT /members/{id}` endpoint with the `members.update` permission.

## Key Concepts

| Concept | Description |
|---|---|
| `auth_methods` | The methods a membership is **allowed** to use. Empty array = all methods allowed. |
| `available_methods` | All methods the platform supports (`password`, `oauth2`, `magic_link`, `webauthn`). |
| `user_enrolled` | Which methods the user has **actually set up** (e.g., has a password hash, has linked OAuth2). |
| Lockout protection | The API prevents removing all enrolled methods — the user must always have at least one way to log in. |
| `default_auth_methods` | Tenant-level setting that controls what new memberships start with (default: `["password"]`). |

## Get Auth Methods

View a member's allowed methods, the platform-supported methods, and what the user has actually enrolled.

**`GET /api/v1/tenants/{tenant_id}/members/{member_id}/auth-methods`**

**Response**

```json
{
  "auth_methods": ["password", "oauth2"],
  "available_methods": ["password", "oauth2", "magic_link", "webauthn"],
  "user_enrolled": [
    { "method": "password", "enrolled": true },
    { "method": "oauth2", "enrolled": true, "details": "accounts.google.com" },
    { "method": "magic_link", "enrolled": true, "details": "john@acme.com" },
    { "method": "webauthn", "enrolled": false }
  ]
}
```

The `details` field provides method-specific context: the OAuth2 issuer URL, the email used for magic links, or the credential name for WebAuthn.

## Update Auth Methods

Replace the allowed authentication methods for a membership. The update takes effect immediately — the next login attempt will be checked against the new list.

**`PUT /api/v1/tenants/{tenant_id}/members/{member_id}/auth-methods`**

| Field | Type | Required | Description |
|---|---|---|---|
| `auth_methods` | string[] | yes | List of allowed methods. Empty array = all methods allowed. |

**Validation rules:**
- Each method must be one of: `password`, `oauth2`, `magic_link`, `webauthn`
- Cannot remove all enrolled methods (would lock the user out)
- Empty array is valid and means "allow all methods" (reset to default behavior)

**Response** — returns the same shape as GET, reflecting the new state.

```json
{
  "auth_methods": ["password", "oauth2"],
  "available_methods": ["password", "oauth2", "magic_link", "webauthn"],
  "user_enrolled": [
    { "method": "password", "enrolled": true },
    { "method": "oauth2", "enrolled": true, "details": "accounts.google.com" },
    { "method": "magic_link", "enrolled": true, "details": "john@acme.com" },
    { "method": "webauthn", "enrolled": false }
  ]
}
```

## Update via General Member Endpoint

Auth methods can also be updated as part of a general membership update. This is useful when changing role and auth methods together.

**`PUT /api/v1/tenants/{tenant_id}/members/{member_id}`**

Pass `auth_methods` alongside other fields. Omitting `auth_methods` (or passing `null`) leaves them unchanged. The same validation and lockout protection apply.

```bash
curl -X PUT http://localhost:8080/api/v1/tenants/ten_acme/members/mem_john-doe_admin \
  -H "Authorization: Bearer $ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "role_id": "role_manager",
    "auth_methods": ["password", "oauth2"]
  }'
```

## Use Cases

### Enforce SSO for managers

After rolling out SSO, restrict all managers to OAuth2 only:

```bash
# First, verify the member has enrolled OAuth2
curl http://localhost:8080/api/v1/tenants/ten_acme/members/mem_jane-doe_manager/auth-methods \
  -H "Authorization: Bearer $ACCESS_TOKEN"

# If oauth2 is enrolled, restrict to it
curl -X PUT http://localhost:8080/api/v1/tenants/ten_acme/members/mem_jane-doe_manager/auth-methods \
  -H "Authorization: Bearer $ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{ "auth_methods": ["oauth2"] }'
```

Jane's next login must use SSO. Password login will be rejected.

### Security incident response

A member's password may be compromised. Restrict them to magic link while investigating:

```bash
curl -X PUT http://localhost:8080/api/v1/tenants/ten_acme/members/mem_bob_mechanic/auth-methods \
  -H "Authorization: Bearer $ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{ "auth_methods": ["magic_link"] }'
```

Bob can still log in via email link, but password and OAuth2 are blocked immediately.

### Gradual onboarding

Invite new members with password only, then enable OAuth2 once SSO is configured:

```bash
# 1. Member is invited — starts with default_auth_methods (["password"])

# 2. After SSO setup, add OAuth2
curl -X PUT http://localhost:8080/api/v1/tenants/ten_acme/members/mem_alice_viewer/auth-methods \
  -H "Authorization: Bearer $ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{ "auth_methods": ["password", "oauth2"] }'

# 3. Once the user has enrolled OAuth2, restrict to SSO only
curl -X PUT http://localhost:8080/api/v1/tenants/ten_acme/members/mem_alice_viewer/auth-methods \
  -H "Authorization: Bearer $ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{ "auth_methods": ["oauth2"] }'
```

### Reset restrictions

Remove all restrictions and allow any enrolled method:

```bash
curl -X PUT http://localhost:8080/api/v1/tenants/ten_acme/members/mem_bob_mechanic/auth-methods \
  -H "Authorization: Bearer $ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{ "auth_methods": [] }'
```

## Configure Tenant Defaults

New memberships (created directly or via invite acceptance) use the `default_auth_methods` tenant setting. Update it to change what methods new members start with.

**`PUT /api/v1/tenants/{tenant_id}/settings/default_auth_methods?module=members`**

```bash
curl -X PUT "http://localhost:8080/api/v1/tenants/ten_acme/settings/default_auth_methods?module=members" \
  -H "Authorization: Bearer $ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "value": "[\"password\",\"oauth2\"]"
  }'
```

After this, all new members and accepted invites in this tenant will start with `["password", "oauth2"]`.

| Default value | Behavior |
|---|---|
| `["password"]` | New members can only use password (platform default) |
| `["password", "oauth2"]` | New members can use password or OAuth2 |
| `[]` | New members can use any enrolled method |

## Common Errors

| Status | Code | Description |
|---|---|---|
| `400` | `MEMBERS.VALIDATION.UNKNOWN_AUTH_METHOD` | A method in the request is not one of `password`, `oauth2`, `magic_link`, `webauthn`. |
| `403` | `CORE.AUTHORIZATION.FORBIDDEN` | The caller lacks the `members.manage_auth` permission (dedicated endpoints) or `members.update` (general endpoint). |
| `404` | `CORE.NOT_FOUND.MEMBER` | The membership ID does not exist in this tenant. |
| `422` | `MEMBERS.VALIDATION.WOULD_LOCK_OUT_USER` | The update would remove all methods the user has enrolled, locking them out. |

**Lockout error example:**

```json
{
  "code": "MEMBERS.VALIDATION.WOULD_LOCK_OUT_USER",
  "message": "cannot remove all enrolled auth methods — user would be locked out",
  "status": 422
}
```

This occurs when the new `auth_methods` list doesn't include any method the user has actually set up. For example, setting `["webauthn"]` when the user has only enrolled `password` and `oauth2`.

## How Enforcement Works

When a user logs in, the `AuthManager.Login()` function checks:

1. Resolve the user's membership for the requested tenant
2. Read the `auth_methods` array from the membership
3. If the array is non-empty, verify the login method is in the list
4. If not allowed, reject with `CORE.AUTHENTICATION.METHOD_NOT_ALLOWED`

This check happens on every login — changes take effect immediately with no restart or cache invalidation.
