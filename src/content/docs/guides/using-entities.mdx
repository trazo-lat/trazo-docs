import { Aside } from '@astrojs/starlight/components';

The `entities` module is a foundation module that provides a configurable, per-tenant type system for real-world objects: vehicles, customers, patients, products, suppliers, and any custom types a tenant defines. Domain modules use entities as input by referencing them by `(type_name, public_id)` and resolving the `EntitiesService` at runtime.

This guide shows the full surface a domain module can use, with working examples drawn from the platform's own modules.

---

## When to Use Entities (and When Not To)

| Use entities when... | Use your own table when... |
|---|---|
| The shape of the record is tenant-configurable (custom fields per tenant) | The shape is fixed and uniform across tenants |
| Multiple modules need to reference the same record (customer, vehicle, ...) | The record is internal to one module's domain |
| You want CRUD, search, audit, permissions, and i18n for free | You need foreign-key constraints across rows the database enforces |
| Tenants must be able to install or define the type via the UI | The structure is part of the platform's contract |

A domain module that needs to **reference** an entity (a billing module referencing a `customer`, a repairs module referencing a `vehicle`) consumes entities. A module whose own data has fixed structure (an `invoice`, a `repair`, a `message`) owns its own table and references entities by ID.

---

## Declaring the Dependency

Declare `entities` as a dependency in your `module.yaml`. The platform validates that the dependency is enabled at startup and refuses to load your module otherwise.

```yaml
# internal/modules/domain/<your-module>/module.yaml
name: your-module
version: "1.0.0"
type: domain

dependencies:
  - entities
```

In `Init`, fail fast if the dependency is missing:

```go
func (m *Module) Init(c *core.Core) error {
    m.core = c

    if !c.Modules.Enabled(context.TODO(), "entities") {
        return fmt.Errorf("your-module: required dependency %q not enabled", "entities")
    }

    // ... continue init
    return nil
}
```

## Resolving the EntitiesService

The entities module registers its service via `core.RegisterService[service.EntitiesService]("entities")`. Consumers retrieve it with the typed accessor.

```go
import (
    entitiessvc "platform/internal/modules/foundation/entities/service"
    entitiesmodel "platform/internal/modules/foundation/entities/model"
)

entities, err := core.GetService[entitiessvc.EntitiesService]("entities")
if err != nil {
    return fmt.Errorf("resolving entities service: %w", err)
}
```

<Aside type="tip">
  Resolve the service once during `Init` and store it on your service struct. Repeated lookups work but cost a registry lookup and a type assertion per call.
</Aside>

### Defining a Minimal Local Contract

For testability, prefer defining the slice of the `EntitiesService` interface your module actually needs in a `deps.go` file inside your service package. Resolve and type-assert it at init time. This is the same pattern the entities module itself uses for the users module — see `internal/modules/foundation/entities/service/deps.go`.

```go
// internal/modules/domain/service-orders/service/deps.go
package service

import (
    "context"
    "errors"

    "platform/internal/core"
    entmodel "platform/internal/modules/foundation/entities/model"
)

// entitiesLookup is the slice of EntitiesService that service-orders needs.
type entitiesLookup interface {
    Get(ctx context.Context, typeName entmodel.EntityTypeName, id string) (*entmodel.Entity, error)
    GetMany(ctx context.Context, typeName entmodel.EntityTypeName, ids []string) ([]entmodel.Entity, error)
    GetType(ctx context.Context, typeName entmodel.EntityTypeName) (*entmodel.EntityType, error)
}

var errEntitiesContract = errors.New("entities module does not satisfy lookup contract")

func resolveEntities(c *core.Core) (entitiesLookup, error) {
    svc, err := c.Modules.Service(core.ModuleName("entities"))
    if err != nil {
        return nil, err
    }
    el, ok := svc.(entitiesLookup)
    if !ok {
        return nil, errEntitiesContract
    }
    return el, nil
}
```

This keeps your module testable without an entities-package import — see [Testing Modules that Consume Entities](#testing-modules-that-consume-entities) below.

## Referencing an Entity as Input

Most domain modules store an entity reference as a string field on their own row. The platform convention is to store the **public ID** (e.g. `cust_acme-corp`, `veh_toyota-camry-2024`) rather than the internal UUID — public IDs are stable, human-readable, and safe to expose in API responses.

```go
type CreateRepairInput struct {
    VehicleID   string `json:"vehicle_id"  doc:"Public ID of the vehicle entity"`
    CustomerID  string `json:"customer_id" doc:"Public ID of the customer entity"`
    Description string `json:"description"`
}
```

Inside the service, validate that the referenced entities exist before persisting:

```go
func (s *Service) Create(ctx context.Context, input CreateRepairInput) (*Repair, error) {
    // Step 1 — Validate input shape
    if input.VehicleID == "" {
        return nil, s.core.Errors.Validation("vehicle_id", "is required")
    }
    if input.CustomerID == "" {
        return nil, s.core.Errors.Validation("customer_id", "is required")
    }

    // Step 2 — Check permission
    if err := s.core.Permission.MustCan(ctx, "repairs.create"); err != nil {
        return nil, err
    }

    // Validate FK references via the entities module
    vehicle, err := s.entities.Get(ctx, "vehicle", input.VehicleID)
    if err != nil {
        return nil, fmt.Errorf("validating vehicle: %w", err)
    }
    if vehicle == nil {
        return nil, s.core.Errors.NotFound("vehicle", input.VehicleID)
    }

    customer, err := s.entities.Get(ctx, "customer", input.CustomerID)
    if err != nil {
        return nil, fmt.Errorf("validating customer: %w", err)
    }
    if customer == nil {
        return nil, s.core.Errors.NotFound("customer", input.CustomerID)
    }

    // ... continue with the standard 5-step pattern
}
```

<Aside type="note">
  The entities service already applies tenant scope and permission checks. A `Get` that returns `nil` means the entity does not exist *or* the caller cannot read it — either way, the safe response is a 404.
</Aside>

<Aside type="caution">
  The example above hardcodes `"vehicle"` and `"customer"`, which only fits a module dedicated to one type. If your module is generic — service orders, tickets, assignments — and different tenants target different types (auto shops service vehicles, clinics service patients), read [Targeting a Tenant-Configurable Entity Type](#targeting-a-tenant-configurable-entity-type) below.
</Aside>

## Reading Entity Data

The `Entity` model carries fields and tenant-defined values in a `Data` JSONB blob. Field types are declared in the entity type's `Schema.Fields` and validated at write time, so reads can rely on the shape but should still handle missing optional fields defensively.

```go
// model.Entity (excerpt)
type Entity struct {
    ID       EntityID       // public ID, e.g. "cust_acme"
    TypeName EntityTypeName // e.g. "customer"
    Data     core.JSONB     // map[string]any of typed fields
    UserID   *string        // set when linked to a user account
    // ... timestamps, audit fields
}
```

Read a single field from the JSONB blob:

```go
customer, err := s.entities.Get(ctx, "customer", input.CustomerID)
if err != nil || customer == nil {
    return nil, s.core.Errors.NotFound("customer", input.CustomerID)
}

email, _ := customer.Data["email"].(string)
taxID, _ := customer.Data["tax_id"].(string)

invoice.BillTo = BillToAddress{
    Name:  fmt.Sprint(customer.Data["name"]),
    Email: email,
    TaxID: taxID,
}
```

<Aside type="caution">
  Always type-assert JSONB values with the comma-ok form. A field declared as `decimal` in the schema arrives as `float64`; an `integer` arrives as `float64` (JSON does not distinguish); a `select` arrives as `string`. Reading the schema via `entities.GetType` is the only way to know the exact intent — see [Discovering the Schema](#discovering-the-schema) below.
</Aside>

### Bulk Reads for List Views

When rendering a list of your own rows that each reference an entity, batch the lookups with `GetMany` rather than calling `Get` in a loop:

```go
// Collect unique customer IDs from a page of invoices
customerIDs := make([]string, 0, len(invoices))
seen := map[string]bool{}
for _, inv := range invoices {
    if !seen[inv.CustomerID] {
        seen[inv.CustomerID] = true
        customerIDs = append(customerIDs, inv.CustomerID)
    }
}

customers, err := s.entities.GetMany(ctx, "customer", customerIDs)
if err != nil {
    return nil, fmt.Errorf("loading customers: %w", err)
}

byID := make(map[string]*entmodel.Entity, len(customers))
for i := range customers {
    byID[string(customers[i].ID)] = &customers[i]
}

// Decorate each invoice. Missing customers (deleted or not visible to the
// caller) keep the empty CustomerName — render a placeholder in the UI,
// do not fail the list.
for i := range invoices {
    if c := byID[invoices[i].CustomerID]; c != nil {
        invoices[i].CustomerName, _ = c.Data["name"].(string)
    }
}
```

### Listing and Searching Entities

For browsing or autocomplete inside your module's UI:

```go
// Paginated list of one type
result, total, err := s.entities.List(ctx, "customer", entmodel.ListEntitiesParams{
    Search: "acme",
    Pagination: core.Pagination{Page: 1, PageSize: 25},
})

// Cross-type search powered by the platform search index
results, total, err := s.entities.Search(ctx, entmodel.SearchParams{
    TypeName: "vehicle", // optional; omit to search all types
    Query:    "toyota camry",
    Pagination: core.Pagination{Page: 1, PageSize: 10},
})
```

## Discovering the Schema

Modules that render dynamic forms or validate input against the tenant's configured fields read the type definition:

```go
typ, err := s.entities.GetType(ctx, "customer")
if err != nil {
    return nil, err
}

for _, field := range typ.Schema.Fields {
    // field.Name, field.Type ("text", "integer", "select", "image", "file", "document", ...)
    // field.Required, field.Options (for "select"), field.Min/Max, field.Pattern
}

// The display label for an entity instance comes from this field
labelField := typ.Config.LabelField
```

`ListTypes` returns every type the tenant has installed or defined. Useful when your module needs to declare which entity types it can target (a notification template selecting "any customer-like type", for instance), or when populating a tenant-admin dropdown to configure the target type for a generic module.

## Bootstrapping Types for a Tenant

Tenants get entity types into their schema in two ways, then activate them:

```go
// 1. Install from the platform's catalog of built-in templates.
//    Built-ins live in internal/modules/foundation/entities/catalog/*.yaml:
//    vehicle, customer, patient, product, supplier.
created, err := s.entities.InstallFromCatalog(ctx, []entmodel.EntityTypeName{
    "vehicle", "customer",
})

// 2. Or define a custom type from scratch with a tenant-specific schema.
custom, err := s.entities.CreateType(ctx, entmodel.CreateEntityTypeInput{
    TypeName:    "unit",
    DisplayName: "Property Unit",
    Schema: entmodel.EntitySchema{Fields: []entmodel.FieldDef{
        {Name: "address", Type: entmodel.FieldTypeText, Required: true},
        {Name: "floor",   Type: entmodel.FieldTypeInteger},
    }},
    Config: entmodel.EntityTypeConfig{LabelField: "address"},
})

// 3. Activate. Both paths produce types in an inactive state.
//    Activation generates the dynamic per-type permissions
//    (entities.<type>.read.all, entities.<type>.create, ...).
err = s.entities.ActivateType(ctx, "unit")
```

The typical flow runs all three from a tenant-admin API call or CLI command during onboarding. A consumer module that requires a specific type (Pattern B below) should detect its absence and surface a setup instruction — it should **not** silently install types on the tenant's behalf.

## Targeting a Tenant-Configurable Entity Type

Some modules don't know upfront which entity type they target. A "service orders" module for an auto shop services `vehicle` entities; the same module for a clinic services `patient` entities; a property-management tenant might service `unit` entities they defined themselves. The module itself cannot hardcode the type — the tenant chooses.

The platform supports three patterns for this. Pick based on whether one tenant ever services multiple types simultaneously, and how independent the per-type lifecycle is.

### Pattern A — Polymorphic Per-Row

Store both `entity_type` and `entity_id` columns on each of your own rows. Different rows in the same tenant can target different types. The module is type-agnostic at compile time.

**Migration:**

```sql
-- migrations/modules/service_orders.sql
CREATE TABLE IF NOT EXISTS service_orders (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    public_id       TEXT NOT NULL UNIQUE,
    tenant_id       UUID NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
    entity_type     TEXT NOT NULL,
    entity_id       TEXT NOT NULL,
    description     TEXT NOT NULL,
    status          TEXT NOT NULL DEFAULT 'open',
    assigned_to     UUID,
    created_by      UUID NOT NULL,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    deleted_at      TIMESTAMPTZ
);

CREATE INDEX IF NOT EXISTS idx_service_orders_entity
    ON service_orders(entity_type, entity_id)
    WHERE deleted_at IS NULL;
```

There is intentionally no foreign key from `entity_id` to the entities table — see [Cross-module FKs are service-enforced](#cross-module-fks-are-service-enforced) below.

**Input and validation:**

```go
type CreateServiceOrderInput struct {
    EntityType  string `json:"entity_type" doc:"Entity type name, e.g. vehicle or patient"`
    EntityID    string `json:"entity_id"   doc:"Public ID of the target entity"`
    Description string `json:"description"`
}

func (s *Service) Create(ctx context.Context, input CreateServiceOrderInput) (*ServiceOrder, error) {
    if input.EntityType == "" {
        return nil, s.core.Errors.Validation("entity_type", "is required")
    }
    if input.EntityID == "" {
        return nil, s.core.Errors.Validation("entity_id", "is required")
    }

    // Validate the type exists and is active for this tenant
    typ, err := s.entities.GetType(ctx, entmodel.EntityTypeName(input.EntityType))
    if err != nil {
        return nil, err
    }
    if typ == nil || !typ.Active {
        return nil, s.core.Errors.Validation("entity_type", "unknown or inactive")
    }

    // Optional: restrict to a tenant-configured allow-list of types
    allowed := core.SettingStringList(ctx, s.core.Config,
        "service-orders", "allowed_entity_types", nil)
    if len(allowed) > 0 && !slices.Contains(allowed, input.EntityType) {
        return nil, s.core.Errors.Validation("entity_type", "not allowed by tenant settings")
    }

    // Validate the (type, id) pair as a unit. Calling Get with the resolved
    // typ.TypeName is what catches "vehicle id paired with patient public ID":
    // the entities service scopes by type internally, so a wrong-type ID
    // returns nil.
    target, err := s.entities.Get(ctx, typ.TypeName, input.EntityID)
    if err != nil {
        return nil, err
    }
    if target == nil {
        return nil, s.core.Errors.NotFound(input.EntityType, input.EntityID)
    }

    // ... persist with both entity_type and entity_id
}
```

**Allow-list setting in `module.yaml`:**

```yaml
settings:
  - key: allowed_entity_types
    type: string_list
    default: []
    description: Entity types this module may target. Empty list means all active types.
```

**Bulk decoration in list views** — group your rows by `entity_type`, batch per type, log and continue when a batch errors, render a placeholder when an entity is missing or invisible to the caller:

```go
type ServiceOrderView struct {
    ServiceOrder
    EntityLabel string `json:"entity_label,omitempty"`
}

func (s *Service) ListForDisplay(ctx context.Context, filter ListFilter) ([]ServiceOrderView, error) {
    orders, _, err := s.repo.List(ctx, filter)
    if err != nil {
        return nil, err
    }

    // Group entity IDs by type so we can batch lookups per type
    byType := map[string][]string{}
    for _, o := range orders {
        byType[o.EntityType] = append(byType[o.EntityType], o.EntityID)
    }

    // Resolve labels per type. Per-type failures (e.g. a type was deleted
    // entirely) are logged and skipped — they must not blank the whole page.
    labels := map[string]string{} // keyed by "<type>:<id>"
    for typeName, ids := range byType {
        ents, err := s.entities.GetMany(ctx, entmodel.EntityTypeName(typeName), ids)
        if err != nil {
            s.core.Log.Logger(ctx).WithModule("service-orders").
                Warn("loading entities for list view",
                    map[string]any{"type": typeName, "error": err.Error()})
            continue
        }
        typ, _ := s.entities.GetType(ctx, entmodel.EntityTypeName(typeName))
        labelField := "name"
        if typ != nil && typ.Config.LabelField != "" {
            labelField = string(typ.Config.LabelField)
        }
        for i := range ents {
            label, _ := ents[i].Data[labelField].(string)
            labels[typeName+":"+string(ents[i].ID)] = label
        }
    }

    views := make([]ServiceOrderView, len(orders))
    for i, o := range orders {
        label := labels[o.EntityType+":"+o.EntityID]
        if label == "" {
            // Entity was soft-deleted, type was removed, or the caller lacks
            // read permission on the entity. The order itself is still
            // visible — render a placeholder, do not omit the row.
            label = "(unavailable)"
        }
        views[i] = ServiceOrderView{ServiceOrder: o, EntityLabel: label}
    }
    return views, nil
}
```

<Aside type="tip">
  Pattern A is the right answer for "service orders", "tickets", "assignments", "tasks", and other modules where the target can vary per row even within one tenant.
</Aside>

### Pattern B — Single Tenant-Configured Type

When a given tenant always targets exactly one type — set during onboarding and rarely changed — store the type as a tenant setting and only the entity ID on each row. Your migration stays simple.

**`module.yaml`:**

```yaml
settings:
  - key: target_entity_type
    type: string
    default: ""
    description: |
      The entity type this module targets for this tenant
      (e.g. "vehicle" for an auto shop, "patient" for a clinic).
```

**Service:**

```go
func (s *Service) Create(ctx context.Context, input CreateInput) (*Order, error) {
    targetType := core.SettingString(ctx, s.core.Config,
        "service-orders", "target_entity_type", "")
    if targetType == "" {
        return nil, s.core.Errors.Module(
            "SERVICE_ORDERS.CONFIG.MISSING_TARGET_TYPE", 412,
            "Set target_entity_type before creating service orders.")
    }

    target, err := s.entities.Get(ctx, entmodel.EntityTypeName(targetType), input.EntityID)
    if err != nil || target == nil {
        return nil, s.core.Errors.NotFound(targetType, input.EntityID)
    }
    // ... persist with just entity_id (type is implicit from the setting)
}
```

<Aside type="caution">
  Changing the setting after rows exist invalidates the existing rows' references — they now point into a different type's namespace. See [Pattern B: setting rotation](#pattern-b-setting-rotation) below for the lock-or-snapshot mitigations.
</Aside>

### Pattern C — Multi-Instance Modules

The platform's multi-instance system creates separate tables, permissions, and events per instance. An auto shop tenant installs `service-orders-vehicles`; a clinic installs `service-orders-patients`. Each instance carries its own `target_entity_type` config and runs independently.

See [Multi-Instance Modules](/architecture/multi-instance-modules/). Choose this when:

- Permissions, status machines, or settings must differ by type (e.g. patients need HIPAA-flavored workflows that vehicles don't)
- Cross-instance reporting is acceptable via wildcard event subscriptions (`service-orders.*.created`)
- The lifecycle of orders against vehicles is materially different from orders against patients

### Choosing Between Them

| If... | Use |
|---|---|
| One tenant services exactly one type, decided at install time | **B** (setting) |
| One tenant services multiple types, mixed in the same list view | **A** (polymorphic) |
| Each type needs different permissions, status machines, or events | **C** (multi-instance) |

## Operational Considerations

The patterns above introduce risk surfaces that the platform does not fix automatically. Each must be handled by the consuming module. Skipping these turns "we depend on entities" into "we have a graveyard of broken references."

### Orphaned references when entities are soft-deleted

Entities are soft-deleted: `Delete` sets `deleted_at` on the row, and subsequent `Get`/`GetMany` calls return nil for it. Without action, your module's rows now point at vanished entities.

Subscribe to `entities.deleted` and pick a cleanup strategy:

```yaml
# module.yaml
events:
  subscribes:
    - event: entities.deleted
      handler: OnEntityDeleted
      async: true
```

| Strategy | Use when | Implementation |
|---|---|---|
| **Detach** | Row has value without the entity (a closed service order is a historical record) | Clear the FK column, optionally set a status flag, keep the row |
| **Cascade soft-delete** | Row is meaningless without the entity (a draft with no other context) | Soft-delete your row in the same handler |

```go
func (s *Service) OnEntityDeleted(ctx context.Context, ev core.Event) error {
    typeName, _ := ev.Payload["type_name"].(string)
    publicID, _ := ev.Payload["public_id"].(string)
    // Idempotent: re-running the same (type, id) detach is a no-op.
    return s.repo.MarkEntityDetached(ctx, typeName, publicID)
}
```

<Aside type="caution">
  **You cannot block the deletion synchronously from your module's Go code.** The `entities.before_delete` hook is a [tenant-configurable webhook or expression](/architecture/hooks/), not a Go-side handler registration. Per the architecture: "Hooks are best-effort — errors are logged but never block the business operation." Even a tenant's webhook that returns an error cannot reject the delete; the error surfaces in `result.Errors` for visibility only.
</Aside>

If "cannot delete entity while X references it" is a hard requirement, enforce it at the **UX layer** instead:

- Your module's API exposes a count or check endpoint (e.g. `GET /api/v1/service-orders/by-entity/{type}/{id}/count`) that the entity-delete UI calls before allowing the action.
- The entity-admin UI displays the dependent rows and requires the user to close or transfer them first.
- A tenant who wants a stricter guard can configure a `entities.before_delete` webhook that calls your count endpoint and surfaces the issue in `result.Errors` — still best-effort, but at least visible in the entity-admin response.

Without one of these in place, plan on the **Detach** strategy as the default. Orphan references that surface as "(unavailable)" labels are recoverable. A delete that quietly cascades through a poorly-tested handler is not.

### Permission scoping divergence between your module and entities

A user with `service-orders.read.all` is not automatically granted `entities.vehicle.read.all`. The platform's permission scopes are evaluated independently per module. The expected behavior is:

- The user sees every service order they're authorized to read.
- For rows whose entity they cannot read, `GetMany` returns no entity for that ID — `byID[id]` is `nil`.
- The list view renders the order with a placeholder label, not a 403, and not an omitted row.

```go
// Pattern A list-view decoration handles missing entities the same way
// whether they're soft-deleted, permission-filtered, or in a removed type.
label := labels[o.EntityType+":"+o.EntityID]
if label == "" {
    label = "(unavailable)"
}
```

<Aside type="caution">
  Do not propagate per-entity errors or permission denials into a 4xx/5xx on your module's list endpoint. The user has permission to see the rows; the missing labels are a symptom, not a failure.
</Aside>

### Deactivated types

A tenant can deactivate a type via `entities.DeactivateType`. The type's existing entities remain readable; new `Create` calls fail with `ENTITIES.VALIDATION.TYPE_INACTIVE`. Your module's behavior:

- **Create**: check `typ.Active` and reject. Already shown in Pattern A's `Create`.
- **Read**: keep working. Your list view shouldn't break because the tenant paused new entity creation.
- **Update**: typically follows Create — if you wouldn't accept new orders against an inactive type, don't accept edits to existing ones either, unless the edit is closing the order.

Listen for `entities.type.deactivated` if you want to surface a warning banner in your UI for tenants ("Type 'vehicle' is inactive — open orders can still be processed").

### Race between validation and write

Between `s.entities.Get(...)` and your own `INSERT`, the type could be deactivated or the entity could be soft-deleted. The window is narrow but real.

- **Best-effort** (default): validate outside the transaction. The next read returns the orphan-handling fallback. Cost: occasional rows pointing at just-deleted entities; correct behavior after `entities.deleted` is handled.
- **Strict**: re-validate inside the transaction. One extra cross-module call per write. Use when the row carries financial or compliance weight (an invoice, a regulated record).

```go
// Strict variant — re-fetch inside the tx
err = s.core.DB.Tx(ctx, func(tx core.Tx) error {
    target, _ := s.entities.Get(ctx, typ.TypeName, input.EntityID)
    if target == nil {
        return s.core.Errors.NotFound(input.EntityType, input.EntityID)
    }
    return s.repo.CreateTx(ctx, tx, input)
})
```

<Aside type="note">
  `s.entities.Get` does not join your transaction — it's a cross-module call. Strict re-validation closes the entity-deletion race but cannot close the type-deactivation race atomically. The platform offers no synchronous cross-module veto mechanism for this case (hooks are tenant-side and best-effort).
</Aside>

### Pattern B: setting rotation

If a tenant changes `target_entity_type` after rows exist, every `entity_id` now refers to a different type's namespace. There is no automatic guard. Two mitigations:

```go
// 1. Lock the setting once data exists. Wire this into your admin endpoint
//    that updates the setting.
func (s *Service) ValidateTargetTypeChange(ctx context.Context, newType string) error {
    current := core.SettingString(ctx, s.core.Config,
        "service-orders", "target_entity_type", "")
    if newType == current {
        return nil
    }
    count, _ := s.repo.RowCount(ctx)
    if count > 0 {
        return s.core.Errors.Module(
            "SERVICE_ORDERS.CONFIG.TYPE_LOCKED", 409,
            "Cannot change target_entity_type while service orders exist. Migrate first.")
    }
    return nil
}
```

```sql
-- 2. Snapshot the type onto each row at insert time. At which point you've
--    rebuilt Pattern A — promote to it.
ALTER TABLE service_orders ADD COLUMN entity_type TEXT NOT NULL DEFAULT '';
```

If you find yourself adding option 2, the tenant has outgrown Pattern B. Migrate to Pattern A.

### Don't parse public-ID prefixes

Public IDs look like `cust_acme-corp`, `veh_toyota-camry-2024`, `pat_jane-doe`. The prefix is **tenant-configurable** via `EntityTypeConfig.PublicID.Prefix` — one tenant can configure `customer` as `client_*` and another as `cust_*`. Both are valid `customer` entities.

Always carry the explicit `entity_type` (Pattern A column, Pattern B setting, Pattern C instance config). Never infer the type from the prefix.

### Cross-module FKs are service-enforced

`entity_id TEXT` has no `REFERENCES entities(public_id)` clause. Cross-module foreign keys aren't enforced at the database level because:

- The entities table lives in the entities module's tables — you don't own it.
- The entities module scopes reads by permission. A DB-level FK would happily join across permission boundaries.
- Soft-delete is opaque to the database: a row with `deleted_at` is still present in the table.

Referential integrity for entity references is enforced at the **service layer** (via the `entities.Get`/`GetMany` API and the `entities.deleted` subscription), not by the database. Plan for it.

## Managing Relations

Relations are first-class. A relation definition describes how two entity types relate (`vehicle` `belongs_to` `customer`), and relation instances connect specific entities.

```go
// Create the definition once (typically as part of tenant setup)
rd, err := s.entities.CreateRelationDef(ctx, entmodel.CreateRelationDefInput{
    Name:       "owner",
    SourceType: "vehicle",
    TargetType: "customer",
    Kind:       entmodel.RelationBelongsTo,
    Required:   true,
})

// Link two specific entities
rel, err := s.entities.CreateRelation(ctx, "vehicle", "veh_toyota-camry-2024",
    entmodel.CreateRelationInput{
        DefinitionID: string(rd.ID),
        TargetID:     "cust_acme-corp",
        Metadata: map[string]any{
            "since": "2024-01-15",
        },
    },
)

// Read all relations for an entity (used when rendering a "Related" panel)
relations, err := s.entities.ListRelations(ctx, "vehicle", "veh_toyota-camry-2024")
```

<Aside type="tip">
  Relations live in the entities module — they don't appear in your module's tables. If you want a hard foreign key on your own row (e.g. an invoice that *must* point to one customer), store the public ID directly on the invoice. Use relations when the link is many-to-many, optional, carries metadata, or is configurable per tenant.
</Aside>

## User-Linked Entities

When a type sets `config.user_linkable: true`, an entity instance can be bound to a platform user account. This is how a `customer` or `patient` entity becomes the "principal" record for someone logging into the tenant.

```go
// Link a customer entity to a user
ent, err := s.entities.LinkUser(ctx, "customer", "cust_acme",
    entmodel.LinkUserInput{UserID: "usr_jane-doe"},
)

// Resolve the user behind an entity (returns *core.User)
user, err := s.entities.GetLinkedUser(ctx, "customer", "cust_acme")

// From the user's session: list their linked entities across types
mine, err := s.entities.ListMyEntities(ctx)
```

A module that targets a specific user (sending a notification, scoping a query) can route by the linked entity rather than by user ID, which keeps tenant-facing IDs stable even if the underlying user account changes.

## Reacting to Entity Lifecycle Events

The entities module emits domain events for every type and instance mutation. Subscribe in your `module.yaml` and handle them via the event bus.

| Event | When it fires |
|---|---|
| `entities.type.created` | A new entity type is created (tenant-defined or installed from catalog) |
| `entities.type.activated` / `entities.type.deactivated` | Type is enabled or disabled for CRUD |
| `entities.created` / `entities.updated` / `entities.deleted` | An entity instance is created, updated, or soft-deleted |
| `entities.relation.created` / `entities.relation.deleted` | An entity relation is created or removed |
| `entities.user.linked` / `entities.user.unlinked` | A user account is linked or unlinked from an entity |

<Aside type="caution">
  Event handlers must be idempotent. The outbox processor retries on failure, and the same event may arrive more than once. A naturally idempotent `UPDATE ... WHERE entity_type=$1 AND entity_id=$2` is the simplest form. When the side effect isn't naturally idempotent, use `ev.ID` as an idempotency key.
</Aside>

The entities module also exposes **hooks** — `entities.before_create`, `entities.before_update`, `entities.before_delete`, `entities.after_create`, `entities.after_update` — but these are [tenant-configurable webhooks/expressions](/architecture/hooks/), not module subscriptions. Tenants use them to inject business logic without code changes; other modules use **events** (above) for cross-module coordination.

## Testing Modules that Consume Entities

The local-contract pattern (`entitiesLookup` interface in `deps.go`) keeps unit tests cheap. Mock the contract with testify, no entities-module import needed.

Expose a test-only constructor that takes `entitiesLookup` directly — the production `New` resolves it via `resolveEntities(c)`:

```go
// internal/modules/domain/service-orders/service/service.go
func New(c *core.Core, repo Repository) *Service {
    entities, err := resolveEntities(c)
    if err != nil {
        // surfaced at module Init; service constructor stays simple
        return nil
    }
    return &Service{core: c, repo: repo, entities: entities}
}

// NewWithDeps is for tests. It bypasses the registry lookup.
func NewWithDeps(c *core.Core, repo Repository, entities entitiesLookup) *Service {
    return &Service{core: c, repo: repo, entities: entities}
}
```

A testify mock satisfying `entitiesLookup`:

## Worked Example: Service Orders (Polymorphic Per-Row)

A "service orders" module needs to work for an auto shop (target: `vehicle`), a clinic (target: `patient`), and a property-management tenant (target: a custom `unit` type they defined). Pattern A keeps the module shippable as-is to all three.

**`module.yaml`:**

```yaml
name: service-orders
type: domain
dependencies:
  - entities

settings:
  - key: allowed_entity_types
    type: string_list
    default: []
    description: Entity types this tenant may target. Empty means all active types.

permissions:
  - key: service-orders.read.all
  - key: service-orders.create
  - key: service-orders.update.all

events:
  subscribes:
    - event: entities.deleted
      handler: OnEntityDeleted
      async: true
```

**Tenant configuration examples:**

| Tenant | `allowed_entity_types` setting |
|---|---|
| `auto-shop-tenant` | `["vehicle"]` |
| `clinic-tenant` | `["patient"]` |
| `mixed-services-tenant` | `["vehicle", "appliance", "boat"]` |
| `property-mgmt-tenant` (custom type) | `["unit"]` |

Each tenant sees the same module, the same API, the same UI — but the dropdown of allowed target types is driven entirely by their settings. The `Create` and `ListForDisplay` implementations are shown under [Pattern A](#pattern-a--polymorphic-per-row). The `OnEntityDeleted` handler uses the **Detach** strategy.

## Worked Example: Billing with a Tenant-Defined Customer Type

The billing module is the inverse case: it always targets `customer`, but the *shape* of `customer` varies by tenant. Some tenants store `tax_id`, some `vat_number`, some both. Billing reads the schema and pulls the field by name.

```go
type Service struct {
    core     *core.Core
    repo     *Repository
    entities entitiesLookup // injected via deps.go
}

func (s *Service) IssueInvoice(ctx context.Context, invoiceID string) (*Invoice, error) {
    inv, err := s.repo.Get(ctx, invoiceID)
    if err != nil {
        return nil, err
    }

    customer, err := s.entities.Get(ctx, "customer", inv.CustomerID)
    if err != nil || customer == nil {
        return nil, s.core.Errors.NotFound("customer", inv.CustomerID)
    }

    // Schema-aware field resolution: pick the tenant's configured tax field
    typ, err := s.entities.GetType(ctx, "customer")
    if err != nil {
        return nil, err
    }
    taxFieldName := "tax_id"
    for _, f := range typ.Schema.Fields {
        if f.Name == "vat_number" {
            taxFieldName = "vat_number"
            break
        }
    }

    inv.BillTo = BillToAddress{
        Name:    fmt.Sprint(customer.Data["name"]),
        Email:   stringField(customer.Data, "email"),
        Address: stringField(customer.Data, "address"),
        TaxID:   stringField(customer.Data, taxFieldName),
    }

    // ... continue issuing
    return inv, nil
}

func stringField(data map[string]any, key string) string {
    v, _ := data[key].(string)
    return v
}
```

Billing hardcodes `"customer"` because every billing tenant has a customer concept — that's the contract billing requires. The customer's *fields* are tenant-configurable; the *type name* is not. Contrast with service orders, where even the type name is tenant-configurable.

## Anti-Patterns

| Anti-Pattern | Why It Breaks |
|---|---|
| Hardcoding a single entity type in a generic module | Different tenants need different target types — use Pattern A, B, or C |
| Expecting hooks (`entities.before_delete` etc.) to block operations from your Go code | Hooks are tenant-configured webhooks/expressions, best-effort by design. They cannot reject an operation and cannot be subscribed to from another module's Go code. Use events for coordination and UX-layer guards for blocking |
| Parsing the `cust_*` / `veh_*` prefix to infer the entity type | Prefixes are tenant-configurable via `EntityTypeConfig.PublicID.Prefix`; always carry an explicit `entity_type` |
| Storing the entity's internal UUID instead of the public ID | Public IDs are the stable, human-readable contract; internal UUIDs leak implementation detail |
| Adding a real DB-level FK from your `entity_id` column to entities | Cross-module FKs aren't enforced; the entities module owns those tables and applies permission scoping. Use the service API and the `entities.deleted` event |
| Treating `entities.deleted` subscription as optional | Without a subscriber, your rows accumulate orphan references that surface as null labels forever |
| Returning a 403 or omitting rows when `GetMany` filters entities by permission | The user has permission to see your row; render a placeholder for the missing entity |
| Failing the entire list view when one entity fetch errors | Log and continue per-type; a partial decoration is better than a blank page |
| Caching `Entity.Data` across requests | Tenants can edit fields at any time; reads should hit the entities service |
| Calling `s.entities.Get` inside a `for` loop over a list page | Use `GetMany` for bulk decoration; group by `entity_type` first when polymorphic |
| Validating `Entity.Data` against a hardcoded shape in your module | Read `Schema.Fields` from `GetType` — the tenant may have changed the type |
| Subscribing to `entities.updated` and writing back into the entity | Creates feedback loops; reach for tenant-configured `entities.before_update` hooks (modify-only, not block) if you need synchronous customization |
| Installing entity types on the tenant's behalf without explicit permission | Bootstrap is a tenant-admin action; consumer modules should detect absence and guide setup |
| Failing your module's `Init` silently when the entities service resolves to `nil` | Make the dependency explicit in `module.yaml` and return a hard error |
| Changing `target_entity_type` (Pattern B) after rows exist without a lock | Existing rows now point into a different type's namespace — lock the setting or snapshot the type per row (which is Pattern A in disguise) |

## Error Codes

The entities module returns typed errors via `core.Error` — defined in `internal/modules/foundation/entities/model/errors.go`. Common codes a consumer module should recognize:

| Code | HTTP | When |
|---|---|---|
| `ENTITIES.NOT_FOUND.TYPE` | 404 | Type does not exist for this tenant |
| `ENTITIES.NOT_FOUND.ENTITY` | 404 | Entity does not exist, or caller cannot read it |
| `ENTITIES.NOT_FOUND.RELATION_DEF` | 404 | Relation definition lookup failed |
| `ENTITIES.NOT_FOUND.RELATION` | 404 | Relation instance lookup failed |
| `ENTITIES.VALIDATION.TYPE_INACTIVE` | 422 | Type is deactivated; cannot create new entities |
| `ENTITIES.VALIDATION.INVALID_DATA` | 422 | Entity data does not match the type schema |
| `ENTITIES.VALIDATION.TYPE_HAS_ENTITIES` | 422 | Attempt to delete a type that still has entities |
| `ENTITIES.VALIDATION.PUBLIC_ID_CONFIG_LOCKED` | 422 | Public ID config cannot change after entities exist |
| `ENTITIES.CONFLICT.DUPLICATE_ENTITY` | 409 | Public ID collision |
| `ENTITIES.CONFLICT.DUPLICATE_RELATION` | 409 | Relation already exists |
| `ENTITIES.VALIDATION.TYPE_NOT_LINKABLE` | 422 | Type does not support user linking |
| `ENTITIES.CONFLICT.ALREADY_LINKED` | 409 | Entity already linked to a user |
| `ENTITIES.CONFLICT.USER_ALREADY_LINKED` | 409 | User already linked to an entity of this type |
| `ENTITIES.VALIDATION.NOT_LINKED` | 422 | Entity is not linked to any user |

In your consumer module, wrap entities errors with `fmt.Errorf("validating %s: %w", typeName, err)` for logging, or translate to your module's namespace when the caller-facing semantic differs — e.g. `ENTITIES.NOT_FOUND.ENTITY` becomes `SERVICE_ORDERS.NOT_FOUND.TARGET` if you want to abstract the dependency from API consumers.

## API Surface at a Glance

The full `EntitiesService` interface lives at `internal/modules/foundation/entities/service/interface.go`. A condensed reference:

| Group | Methods |
|---|---|
| **Catalog** | `ListCatalog`, `InstallFromCatalog` |
| **Types** | `GetType`, `ListTypes`, `CreateType`, `UpdateType`, `ActivateType`, `DeactivateType`, `DeleteType` |
| **Entities** | `Get`, `GetMany`, `List`, `Create`, `Update`, `Delete` |
| **Relations** | `ListRelationDefs`, `CreateRelationDef`, `GetRelationDef`, `UpdateRelationDef`, `DeleteRelationDef`, `ListRelations`, `CreateRelation`, `DeleteRelation` |
| **Search** | `Search` |
| **User linking** | `LinkUser`, `UnlinkUser`, `GetLinkedUser`, `ListMyEntities` |

All entity-level methods (`Get`, `GetMany`, `List`, `Create`, `Update`, `Delete`, `ListRelations`, ...) require an `EntityTypeName` argument. Modules that don't know the type at compile time read it from a setting (Pattern B), from a per-row column (Pattern A), or from their instance config (Pattern C).
