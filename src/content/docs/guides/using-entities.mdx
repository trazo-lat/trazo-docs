This guide is for module authors who need to **reference entities owned by the entities module** — vehicles, customers, patients, or any tenant-defined type. It covers the consumption API: how to resolve a tenant-configured type, validate FK references cheaply, veto destructive operations, and stay agnostic to whatever type your tenant happens to bind.

The architecture rationale and design tradeoffs live alongside the implementation in `core-service/docs/design/entities-module-consumption.md`. This guide is the HOW; that is the WHY.

## The agnostic-module pattern

A platform feature like **work-orders** has no business naming `vehicle` or `customer` in its code. An auto shop tracks work against vehicles; a clinic tracks work against patients; a property manager tracks work against a custom `unit` type. **Same compiled module, three tenants, three semantic meanings.**

Express the dependency as a **role**, not a type name:

The role name (`subject`, `owner`, whatever your module calls it) lives in your code. The actual entity type lives in tenant config. Your service never types the words `vehicle` or `patient`.

### 1. Declare the role

```yaml
# work-orders/module.yaml
name: work-orders
type: domain
dependencies:
  - entities

settings:
  - key: subject_entity_type
    type: string
    default: ""
    description: The entity type this tenant binds to the "subject" role.

entity_roles:
  - name: subject
    description: The entity being serviced — vehicle, patient, table, etc.
    setting_key: subject_entity_type
    required_fields:
      - { name: name, type: text }
      - { name: phone }   # any type accepted (no `type:` set)
```

`required_fields` is your shape contract. List the fields your module actually reads. The tenant can configure additional fields on the bound type — your module just guarantees these will be present.

### 2. Validate at module Init

```go
import entservice "platform/internal/modules/foundation/entities/service"

//go:embed module.yaml
var moduleConfigBytes []byte

func (m *Module) Init(c *core.Core) error {
    m.core = c

    // Fail fast if module.yaml is malformed (missing setting_key, role
    // doesn't reference a declared setting, duplicate names, etc.)
    if err := entservice.ValidateEntityRoles(moduleConfigBytes); err != nil {
        return fmt.Errorf("work-orders: %w", err)
    }

    roles, _ := entservice.ParseEntityRoles(moduleConfigBytes)
    m.subjectRole = roles[0]

    entities, err := core.GetService[entservice.EntitiesService]("entities")
    if err != nil {
        return err
    }
    m.service = NewService(c, entities, m.subjectRole)
    return nil
}
```

<Aside type="tip">
  `ValidateEntityRoles` is yaml-only — it doesn't query the tenant DB. It catches manifest typos (a role pointing at a `setting_key` that no `settings:` entry declares) at startup instead of as a runtime 412 in production.
</Aside>

### 3. Resolve at runtime

```go
func (s *Service) Create(ctx context.Context, input CreateInput) (*Order, error) {
    typ, err := s.entities.ResolveRoleType(ctx, ModuleName, s.subjectRole)
    if err != nil {
        // 412 ENTITIES.ROLE.UNCONFIGURED      — tenant hasn't set the setting
        // 404 ENTITIES.NOT_FOUND.TYPE         — bound type doesn't exist
        // 422 ENTITIES.ROLE.SCHEMA_MISMATCH   — missing required fields
        return nil, err
    }
    if !typ.Active {
        return nil, s.core.Errors.Validation("entity_type", "type is paused")
    }

    // typ.TypeName is the tenant-chosen name; use it for subsequent calls.
    if _, err := s.entities.Get(ctx, typ.TypeName, input.SubjectID); err != nil {
        return nil, err
    }
    // ... persist
}
```

Each `ResolveRoleType` error carries structured `Params` (`module`, `role`, `setting_key`, `missing` fields) so a tenant-admin UI can render an actionable next step instead of a generic 4xx.

## Picking a resolution method

Three different "find an entity" calls, each fastest for its case:

| Method | Returns | Use when |
|---|---|---|
| `Get(typeName, publicID)` | `*Entity` (full JSONB payload), or typed 404 | You need entity data (label, fields, linked user) |
| `Exists(typeName, publicID)` | `(bool, error)` — no row data crossed the wire | You only need to confirm an FK; cheap |
| `GetByField(typeName, fieldName, value)` | `*Entity` matching a non-primary field, or `nil` | Natural-key lookups: find customer by email, vehicle by VIN |
| `GetMany(typeName, ids)` | `[]Entity` — soft-deletes and scope-filtered rows silently omitted | List view decoration; pair with placeholders |

All four honor the same MustCan check (`entities.<type>.read.all` or `.own`) and apply the active scope filter at the SQL level.

<Aside type="caution">
  `GetMany` returning fewer entities than IDs can mean **either** the row was soft-deleted **or** scope filtered it out. The caller can't distinguish — by design, distinguishing would leak permission information. Render a placeholder; don't drop the parent row.
</Aside>

## Strict uniqueness on tenant-defined fields

When an entity type schema flags a field `unique: true`, the platform enforces it in two layers:

<Steps>

1. **Service-level pre-check.** Every `Create` and `Update` runs a `SELECT 1` against the GIN-indexed `data` column. On collision the caller gets a 422 naming the colliding field.

2. **Database partial UNIQUE index.** Created automatically on `CreateType`/`UpdateType` and dropped on `DeleteType`. Predicate: `WHERE type_name = '<type>' AND deleted_at IS NULL AND data ? '<field>'`. Catches concurrent inserts that race past the pre-check.

</Steps>

Flipping `unique: true` on an existing field is rejected if any duplicates already exist — the API returns 422 with the offending count, and the schema change does not commit.

```yaml
# inside an entity type schema (tenant-managed)
fields:
  - name: email
    type: email
    unique: true   # enforced; safe to use with GetByField
```

```go
customer, err := s.entities.GetByField(ctx, "customer", "email", input.Email)
// err == nil && customer == nil → no match
// err == nil && customer != nil → unique match (under read scope)
```

## Pre-mutation guards

The platform exposes four synchronous Go-side guard registries so a consumer module can veto operations that would leave it holding orphan references.

| Guard | Veto target | Lifecycle |
|---|---|---|
| `entities.RegisterDeleteGuard` | Soft-delete of a single entity | Service.Delete, before tenant `before_delete` hook |
| `entities.RegisterTypeDeactivateGuard` | DeactivateType (reversible) | Service.DeactivateType |
| `entities.RegisterTypeDeleteGuard` | DeleteType (covers the "consumer pins type name in config" case) | Service.DeleteType, after the no-entities check |
| `core.Config.RegisterSettingGuard` | Any setting upsert/delete (Pattern B lock) | Settings handler, before `store.Upsert` |

All four follow the same shape: return a non-nil error to abort, filter by the targeted name yourself (registries are global, just keyed by your match), and keep the body fast (one COUNT, no I/O fan-out — guards run on the request path).

### Block entity delete while open refs exist

```go
entities.RegisterDeleteGuard(func(ctx context.Context, ent *entmodel.Entity) error {
    if ent.TypeName != s.targetType { return nil }
    n, err := s.repo.CountOpenForEntity(ctx, string(ent.ID))
    if err != nil { return err }
    if n > 0 {
        return s.core.Errors.Conflict(string(ent.TypeName),
            fmt.Sprintf("cannot delete: %d open work orders", n))
    }
    return nil
})
```

### Lock a Pattern B setting once rows exist

```go
c.Config.RegisterSettingGuard(
    core.SettingGuardMatch{Module: string(ModuleName), Key: string(SettingSubjectType)},
    func(ctx context.Context, change core.SettingChange) error {
        // First-time set or no-op writes always pass.
        if change.OldValue == nil || change.OldValue == change.NewValue {
            return nil
        }
        n, err := s.repo.CountAll(ctx)
        if err != nil { return err }
        if n > 0 {
            return core.NewError("WORK_ORDERS.CONFIG.TYPE_LOCKED",
                fmt.Sprintf("cannot change subject type: %d rows reference the current type", n),
                409, nil)
        }
        return nil
    },
)
```

The `SettingGuardMatch` indexes the registry so guards on other modules' settings don't pay the dispatch cost. Pass `core.SettingGuardMatch{}` (zero match) for a catch-all that fires on every write.

<Aside type="caution">
  Guards are **not** transactional across modules. A concurrent ref-create can race past your `CountOpenForEntity` check. Always pair a guard with a subscription to the corresponding event (`entities.deleted`, `entities.type.deactivated`, etc.) so the orphan that does slip through gets cleaned up after the fact.
</Aside>

## Permission scoping

Entity permission keys are typed per type: `entities.<type>.read.all`, `entities.<type>.update.all`, etc. Grants on your own module do **not** carry across.

A tenant assigning `work-orders.read.all` to a role must also assign `entities.<bound-type>.read.all` for every type the module decorates. Otherwise list endpoints return 403 from the entities side. Document this in your module's install notes.

The check is enforced at the top of every entities service method via `MustCan`. There is no per-entity silent filtering — either the caller holds the perm and `GetMany` returns matching rows, or the call 403s entirely. The "(unavailable)" placeholder in list views handles the soft-delete and scope-removed cases.

## Anti-patterns

| Anti-pattern | Why it breaks | Right way |
|---|---|---|
| Hardcoding `"vehicle"` or `"customer"` in a generic module | Forces a fork per industry | Declare an `entity_role`, resolve at runtime |
| Parsing `cust_*` / `veh_*` prefixes to infer type | Tenant-configurable via `Config.PublicID.Prefix` | Carry `entity_type` alongside `entity_id` |
| Storing `Entity.InternalID` (UUID) on your rows | Implementation detail; can change | Store the public ID |
| Adding `REFERENCES entities(public_id)` | Soft-deletes pass; can't see scope | `Exists` + `entities.deleted` subscription |
| Treating `entities.deleted` as optional when a guard is registered | Guards aren't transactional across modules | Always pair guard with subscription |
| Caching `Entity.Data` across requests | Tenants edit fields anytime | Re-fetch every request |
| `Get` in a `for` loop | N+1 trips | `GetMany`; group by type for polymorphic cases |
| Hardcoding field names without reading `GetType` | Tenant may rename or remove fields | Read schema once at request entry |
| Changing Pattern B target without a lock | Existing `entity_id`s now reference a different namespace | Register a `SettingGuard` that vetoes once rows exist |
| Slow I/O inside any guard | Guards run on the request path | One COUNT/index lookup max; defer heavy work to the async handler |
| Using `Get` to check existence | Loads the full JSONB payload | `Exists` |
