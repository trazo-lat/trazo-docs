Heyllave uses a **declarative, desired-state** migration model powered by `pg-schema-diff`. Each `.sql` file describes what the schema should look like, not the steps to get there. There are no numbered prefixes and no `-- +down` markers — the diff engine computes the actual DDL needed to bring the live database in line with the files at run time.

Migrations are split into three buckets based on **tenant portability**: anything that needs to travel with a tenant lives in the per-tenant tier; cross-tenant data lives in `public`.

---

## The Three Tiers

```
migrations/
├── core/
│   ├── public/        ← global, cross-tenant tables (applied once)
│   └── tenant/        ← per-tenant core subsystem tables (applied to every tenant schema)
└── modules/
    └── <name>.sql     ← per-tenant module tables (applied to every tenant schema)
```

| Tier | Where it lives | What it holds | When applied |
|------|---------------|---------------|--------------|
| `migrations/core/public/` | `public` schema | Cross-tenant registries: tenants, users, memberships, roles, invitations, permission/scope definitions, tenant_modules, migration_history, platform/module translations, platform default notification rules | Once, via the bootstrap endpoint |
| `migrations/core/tenant/` | every `tenant_<slug>` schema | Per-tenant core subsystems: sessions, audit logs, events outbox, event subscriptions/deliveries, tenant_hooks, dashboard_configs, in_app_notifications, tenant_notification_rules, tenant_settings, stored_files, search_index, approval_requests/steps, tenant_state_snapshots, tenant_translations | At tenant creation, for every new tenant |
| `migrations/modules/<name>.sql` | every `tenant_<slug>` schema | Module-owned per-tenant tables (e.g., `agents`, `billing_*`, `conversations`, `documents_*`, `entities`) | At tenant creation for every module listed in `tenant.default_modules`; later via the SchemaProvisioner when a module is enabled |

<Aside type="tip">
The rule of thumb: **tenants must be physically portable**. If a table needs to move when a tenant is exported, restored, or deleted, put it in `core/tenant/` or `modules/`. If it holds cross-tenant data (e.g., the tenant registry itself), put it in `core/public/`.
</Aside>

## File Format

Each migration is a plain SQL file that declares the desired state. No up/down separator, no transactions, no version prefix.

```sql
-- migrations/modules/repairs.sql

CREATE TABLE IF NOT EXISTS repairs (
    id              UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    vehicle_id      UUID         NOT NULL,
    description     TEXT,
    status          VARCHAR(50)  NOT NULL DEFAULT 'pending',
    created_by      UUID         NOT NULL,
    assigned_to     UUID,
    resolution_notes TEXT,
    created_at      TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    deleted_at      TIMESTAMPTZ
);

CREATE INDEX IF NOT EXISTS idx_repairs_status   ON repairs(status);
CREATE INDEX IF NOT EXISTS idx_repairs_assigned ON repairs(assigned_to);
CREATE INDEX IF NOT EXISTS idx_repairs_vehicle  ON repairs(vehicle_id);
```

Notice what is **not** here:

- No `tenant_id` column. Tables in `migrations/modules/` and `migrations/core/tenant/` live inside `tenant_<slug>` — the schema name is the tenancy boundary. Adding a redundant `tenant_id` column would only invite mistakes.
- No `-- +down` block. To remove a column or table, delete it from the SQL file; the next diff will drop it.
- No timestamp prefix in the filename. One file per logical concern: `documents.sql`, `entities.sql`, `agents.sql`, etc.

<Aside type="caution">
Public-schema tables (`migrations/core/public/`) **do** keep their `tenant_id` foreign keys where applicable — for example, `memberships`, `tenant_invitations`, and `service_principals` reference `public.tenants(id)`. That column is meaningful there because the table holds rows for many tenants.
</Aside>

## File Naming

```
migrations/core/public/<topic>.sql
migrations/core/tenant/<topic>.sql
migrations/modules/<module-name>.sql
```

| Tier | Examples |
|------|----------|
| `core/public/` | `tenants.sql`, `users.sql`, `permissions.sql`, `translations.sql`, `notification_rules.sql` |
| `core/tenant/` | `sessions.sql`, `audit_events.sql`, `notifications.sql`, `storage.sql`, `hooks.sql`, `settings.sql` |
| `modules/` | `agents.sql`, `billing.sql`, `conversations.sql`, `documents.sql`, `entities.sql` |

There is exactly **one** file per module. A single module file may declare several related tables (e.g., `documents.sql` declares `document_types`, `document_instances`, `document_signatures`, and `document_share_tokens`).

## How a New Tenant Gets Provisioned

`POST /api/v1/tenants` runs the entire flow inside a single transaction:

<Steps>
1. Insert a row into `public.tenants` with the new tenant's id, slug, and metadata.
2. `CREATE SCHEMA tenant_<slug>`.
3. Set `search_path` on the transaction to `tenant_<slug>, public`.
4. Apply every file in `migrations/core/tenant/` to the new schema.
5. For each module listed in `tenant.default_modules` (from `heyllave.yaml`), apply `migrations/modules/<name>.sql` to the new schema.
6. Seed the default roles (admin, member, etc.) for the tenant.
7. Insert one row per provisioned module into `public.tenant_modules`.
</Steps>

If any step fails, the transaction rolls back: the schema is dropped, the `public.tenants` row is gone, and the system returns to the pre-call state.

The `cmd/seed` binary mirrors this exact flow for local bootstrap. It creates the first tenant and admin user after `POST /api/v1/admin/migrations/bootstrap` has applied the public-schema tables.

## Bootstrapping the Public Schema

The public-schema tables (`tenants`, `users`, `roles`, …) are applied **once** per database, not per tenant. Trigger this via the admin bootstrap endpoint:

```bash
curl -X POST http://localhost:8080/api/v1/admin/migrations/bootstrap \
  -H "Authorization: Bearer $ADMIN_TOKEN"
```

The endpoint diffs `migrations/core/public/` against the live `public` schema and applies whatever DDL is needed to bring it in line. It is idempotent — running it on an already up-to-date database is a no-op.

<Aside type="note">
The bootstrap endpoint does **not** touch any `tenant_<slug>` schemas. Per-tenant migrations run at tenant-creation time (or via the `SchemaProvisioner` when a module is enabled for an existing tenant).
</Aside>

## Enabling a Module on an Existing Tenant

Modules listed in `tenant.default_modules` are provisioned automatically when a tenant is created. To enable a module on an existing tenant, the platform's `SchemaProvisioner` reads `migrations/modules/<name>.sql` and applies it to that tenant's schema only. The flow is:

1. Insert a `tenant_modules` row for `(tenant_id, module_name)` in `public`.
2. Set `search_path` to the tenant's schema.
3. Apply `migrations/modules/<name>.sql`.

The same diff engine ensures the module's tables only land in that one schema.

## CLI Commands

The `heyllave migrate` command surface manages the diff/apply loop and inspects state. The exact command flags are documented under [Migrate Commands](/cli/migrate-commands/); the important conceptual points are:

- `heyllave migrate status` reports which tables are present, missing, or drifted in `public` and in each tenant schema relative to the embedded SQL files.
- `heyllave migrate diff` is read-only: it prints the DDL the engine would run if asked.
- The platform records every applied migration in `public.migration_history` keyed by tier, schema, and file hash.

<Aside type="caution">
Because migrations are declarative, **never** edit a deployed `.sql` file in a way that drops data without intention. Removing a column from the file is taken as "this column should not exist" and the next diff will issue an `ALTER TABLE ... DROP COLUMN`. Treat schema changes the same way you treat any code change: through a PR, with review.
</Aside>

## Writing Good Migrations

### Adding a Table

Just declare it. The diff engine will create it the next time the relevant schema is reconciled.

```sql
CREATE TABLE IF NOT EXISTS parts (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    repair_id   UUID NOT NULL REFERENCES repairs(id) ON DELETE CASCADE,
    name        VARCHAR(255) NOT NULL,
    quantity    INT NOT NULL DEFAULT 1,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### Adding a Column

Add it to the desired-state SQL:

```sql
ALTER TABLE repairs
    ADD COLUMN IF NOT EXISTS sla_deadline TIMESTAMPTZ;

CREATE INDEX IF NOT EXISTS idx_repairs_sla
    ON repairs(sla_deadline)
    WHERE sla_deadline IS NOT NULL AND status NOT IN ('completed', 'cancelled');
```

Or — preferred — express the desired state directly in the table definition and let the diff engine compute the `ALTER TABLE` for you:

```sql
CREATE TABLE IF NOT EXISTS repairs (
    ...
    sla_deadline TIMESTAMPTZ,
    ...
);
```

<Aside type="tip">
Nullable columns with no default can be added to large tables without rewriting the table — PostgreSQL stores them lazily. Avoid `NOT NULL DEFAULT <constant>` on hot tables unless you know the rewrite cost.
</Aside>

### Renaming a Column

Pure renames cannot be inferred from a desired-state diff (the engine cannot tell a rename from a drop+add). Do this as a two-deploy migration:

<Steps>
1. Add the new column alongside the old one. Deploy code that writes to both.
2. Backfill the new column from the old one in a one-off statement.
3. Drop the old column in a follow-up SQL change once nothing reads from it.
</Steps>

### Adding a Status Value

Heyllave stores status values as `VARCHAR(50)` precisely so they can be added without DDL. The validation lives in the application layer and in `module.yaml`'s `statuses` section. No migration is required.

## Cross-Schema References

Inside a `migrations/core/tenant/` or `migrations/modules/` file, bare table names refer to the **tenant schema** (because `search_path` is set to `tenant_<slug>, public` when the file is applied). To point at the public schema, use the fully qualified name:

```sql
-- This refers to tenant_<slug>.repairs because search_path is tenant-scoped.
CREATE TABLE repair_attachments (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    repair_id   UUID NOT NULL REFERENCES repairs(id) ON DELETE CASCADE,
    file_id     UUID NOT NULL  -- references tenant_<slug>.stored_files (also in this schema)
);
```

There is generally no need for a `repair_attachments → public.tenants(id)` foreign key, because every row in `repair_attachments` is implicitly scoped to whichever schema it lives in.

## Embedding at Compile Time

The migration tree is embedded into the server binary:

```go
//go:embed migrations/core/public/*.sql migrations/core/tenant/*.sql migrations/modules/*.sql
var migrations embed.FS
```

This means:

- Binaries are self-contained and reproducible.
- Schema changes are version-controlled alongside the code that depends on them.
- There is no separate "migrations volume" to mount in production.

The platform reads the embedded files at startup, hashes them, and compares the hashes to `public.migration_history` to decide what needs reconciling.
