Trazo uses an Atlas-compatible SQL migration format managed through the `trazo migrate` CLI commands. Migrations are embedded into the module binary at compile time and applied per-tenant schema at runtime.

---

## Migration File Format

Each migration is a single `.sql` file containing the `up` SQL followed by the `down` SQL, separated by the `-- +down` marker.

```sql
-- 20240101000000_create_repairs.sql

CREATE TABLE repairs (
    id          UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id   UUID         NOT NULL REFERENCES public.tenants(id),
    vehicle_id  UUID         NOT NULL,
    description TEXT,
    status      VARCHAR(50)  NOT NULL DEFAULT 'pending',
    created_by  UUID         NOT NULL,
    assigned_to UUID,
    resolution_notes TEXT,
    created_at  TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    deleted_at  TIMESTAMPTZ
);

CREATE INDEX idx_repairs_tenant    ON repairs(tenant_id);
CREATE INDEX idx_repairs_status    ON repairs(status);
CREATE INDEX idx_repairs_assigned  ON repairs(assigned_to);
CREATE INDEX idx_repairs_created   ON repairs(created_by);
CREATE INDEX idx_repairs_vehicle   ON repairs(vehicle_id);

-- +down

DROP TABLE IF EXISTS repairs;
```

### File Naming Convention

```
<timestamp>_<description>.sql
```

| Segment | Format | Example |
|---|---|---|
| `timestamp` | `YYYYMMDDHHMMSS` (UTC) | `20240101120000` |
| `description` | snake_case, brief | `create_repairs` |

The CLI generates the timestamp automatically when you run `trazo migrate create`.

<Aside type="caution">
  Never modify a migration file after it has been applied in any environment. If you need to change the schema, create a new migration. The migration history is stored in a migrations table per schema and checked against file checksums.
</Aside>

## Where Migrations Live

Each module embeds its own migrations. Store all `.sql` files under a `migrations/` subdirectory within the module package:

```
internal/modules/domain/repairs/
├── migrations/
│   ├── 20240101000000_create_repairs.sql
│   ├── 20240215083000_add_sla_deadline.sql
│   └── 20240310140000_add_parts_table.sql
├── module.go
└── ...
```

The `module.go` file embeds the migrations directory:

```go
//go:embed migrations
var migrationFiles embed.FS

func (m *RepairsModule) Migrations() embed.FS {
    return migrationFiles
}
```

The platform applies migrations from all registered modules in dependency order when `trazo migrate up` is run.

## CLI Commands

### Create a New Migration

```bash
trazo migrate create <name>
```

Creates a new migration file with the current UTC timestamp in the module's `migrations/` directory. If multiple modules are present, you will be prompted to select the target module.

```bash
trazo migrate create add_sla_deadline --module repairs
```

Output:

```
Created: internal/modules/domain/repairs/migrations/20240215083000_add_sla_deadline.sql
```

---

### Apply Migrations

```bash
trazo migrate up
```

Applies all pending migrations for all tenants. The platform iterates over every active tenant schema and applies any unapplied migrations in timestamp order.

To apply only a specific module's migrations:

```bash
trazo migrate up --module repairs
```

To apply to a specific tenant schema only:

```bash
trazo migrate up --tenant acme-corp
```

Expected output:

```
Applying migrations for tenant: acme-corp
  [1/3] 20240101000000_create_repairs              OK
  [2/3] 20240215083000_add_sla_deadline            OK
  [3/3] 20240310140000_add_parts_table             OK
All migrations applied for acme-corp.
```

---

### Roll Back Migrations

```bash
trazo migrate down
```

By default, rolls back the most recently applied migration across all tenants.

```bash
# Roll back the last N migrations
trazo migrate down --steps 3

# Roll back a specific migration by name
trazo migrate down --to 20240101000000_create_repairs
```

<Aside type="caution">
  Rolling back in production is a destructive operation. The `-- +down` SQL typically drops tables. Always confirm that a rollback is the correct action and that you have a recent database backup before running `migrate down` on a non-development environment.
</Aside>

### Check Migration Status

```bash
trazo migrate status
```

Lists all migrations for all tenants showing whether each has been applied:

```
Tenant: acme-corp
  [applied] 20240101000000_create_repairs
  [applied] 20240215083000_add_sla_deadline
  [pending] 20240310140000_add_parts_table

Tenant: beta-inc
  [applied] 20240101000000_create_repairs
  [pending] 20240215083000_add_sla_deadline
  [pending] 20240310140000_add_parts_table
```

---

### View Migration History

```bash
trazo migrate history
```

Shows a timestamped log of every migration that has been applied, including who ran it and when:

```
2024-01-01 12:00:00 UTC  20240101000000_create_repairs        applied  (local)
2024-02-15 08:30:00 UTC  20240215083000_add_sla_deadline      applied  (ci-pipeline)
```

---

### Diff Against Current Schema

```bash
trazo migrate diff
```

Compares the state that the migration files would produce against the current live schema and reports any discrepancies. Useful for detecting manual schema changes or missing migrations.

```
Schema drift detected in tenant: acme-corp
  + Column repairs.notes_internal (TEXT) — not in any migration
  Missing index: idx_repairs_vehicle
```

## Writing Good Migrations

### Adding a Column

```sql
-- 20240215083000_add_sla_deadline.sql

ALTER TABLE repairs
    ADD COLUMN sla_deadline TIMESTAMPTZ;

CREATE INDEX idx_repairs_sla ON repairs(sla_deadline)
    WHERE sla_deadline IS NOT NULL AND status NOT IN ('completed', 'cancelled');

-- +down

DROP INDEX IF EXISTS idx_repairs_sla;

ALTER TABLE repairs
    DROP COLUMN IF EXISTS sla_deadline;
```

<Aside type="tip">
  Use `ADD COLUMN ... DEFAULT NULL` (i.e., no DEFAULT, nullable) when adding a column to a large table. PostgreSQL can add a nullable column with no default without rewriting the table, making the migration fast and lock-free.
</Aside>

### Renaming a Column

Rename operations require a multi-step migration to avoid breaking running application code:

<Steps>
1. Add the new column name as a copy of the old one:

   ```sql
   ALTER TABLE repairs ADD COLUMN resolution_notes TEXT;
   UPDATE repairs SET resolution_notes = notes WHERE notes IS NOT NULL;
   ```

2. Deploy application code that writes to both columns.

3. In a second migration, drop the old column:

   ```sql
   ALTER TABLE repairs DROP COLUMN notes;
   ```
</Steps>

### Adding an Enum Value

For `VARCHAR` status columns (Trazo's preferred approach), no migration is needed — status values are validated at the application layer. For `ENUM` types, add the value explicitly:

```sql
ALTER TYPE repair_status ADD VALUE IF NOT EXISTS 'on_hold';

-- +down

-- Note: PostgreSQL does not support removing enum values without recreating the type.
-- Document this limitation in the migration.
```

<Aside type="note">
  Trazo stores status values as `VARCHAR(50)` rather than PostgreSQL `ENUM` types precisely to avoid this migration complexity. Only use `ENUM` if you have a specific reason.
</Aside>

## Multi-Tenant Considerations

Every tenant has its own PostgreSQL schema named `tenant_<slug>`. The `trazo migrate up` command applies each migration to every tenant schema individually.

The `SET search_path` mechanism ensures that unqualified table names in your SQL refer to the tenant schema automatically. You do not need to prefix table names with the schema name.

```sql
-- This refers to `tenant_acme.repairs`, not `public.repairs`
CREATE TABLE repairs (...);
```

To reference platform-level tables (in the `public` schema), always use the fully qualified name:

```sql
-- Cross-schema foreign key
tenant_id UUID NOT NULL REFERENCES public.tenants(id),
```

### New Tenant Provisioning

When a new tenant is created, the platform applies all existing migrations to the new tenant schema in one pass. This means every migration must be idempotent with respect to the schema state (use `IF NOT EXISTS` and `IF EXISTS` guards where possible).

```sql
-- Safe for new tenant provisioning
CREATE TABLE IF NOT EXISTS repairs (...);
CREATE INDEX IF NOT EXISTS idx_repairs_status ON repairs(status);

-- +down
DROP TABLE IF EXISTS repairs;
```

## Embedded Migrations at Compile Time

The `//go:embed migrations` directive in `module.go` embeds the entire `migrations/` directory into the compiled binary. The platform reads migration files from the `embed.FS` at startup, not from the filesystem.

This means:

- No separate migration files need to ship alongside the binary in production.
- Migration files are version-controlled alongside the code that depends on them.
- The exact set of migrations in a binary is deterministic and reproducible.

```go
//go:embed migrations
var migrationFiles embed.FS

func (m *RepairsModule) Migrations() embed.FS {
    return migrationFiles
}
```

The platform iterates over the embedded filesystem, sorts files by name (which sorts by timestamp due to the naming convention), and applies them in order.
