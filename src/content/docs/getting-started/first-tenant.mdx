This walkthrough takes a brand-new Heyllave install from "empty database" to "first authenticated request." The platform uses a two-step provisioning flow:

1. **Bootstrap the public schema** — applies `migrations/core/public/` once per database. Creates `tenants`, `users`, `roles`, and the other cross-tenant tables.
2. **Seed the first tenant** — creates `tenant_<slug>`, applies `migrations/core/tenant/` and the default modules' `migrations/modules/<name>.sql`, then inserts the first admin user, role, and membership.

Make sure the server is running on `localhost:8080` before you begin.

<Aside type="tip">
All examples use `jq` to pretty-print JSON responses. Install it with `apt install jq` or `brew install jq`, or omit the `| jq` pipe if you prefer raw output.
</Aside>

---

<Steps>

1. ## Bootstrap the Public Schema

   The bootstrap endpoint reconciles `migrations/core/public/` against the database. It only touches the `public` schema — no tenant schemas are created here.

   ```bash
   curl -s -X POST http://localhost:8080/api/v1/admin/migrations/bootstrap | jq
   ```

   Response `200 OK`:

   ```json
   {
     "status": "ok",
     "applied": [
       "tenants.sql",
       "users.sql",
       "access_control.sql",
       "permissions.sql",
       "modules.sql",
       "invitations.sql",
       "notification_rules.sql",
       "translations.sql",
       "migration_history.sql"
     ]
   }
   ```

   The endpoint is idempotent — running it on an already up-to-date database is a no-op (the `applied` list will be empty).

   <Aside type="note">
     Bootstrap creates the `public` schema only. It does **not** create any `tenant_<slug>` schemas, and does **not** insert any rows. The next step does both.
   </Aside>

2. ## Seed the First Tenant and Admin

   Run the `cmd/seed` binary. It connects using the database URL from `heyllave.yaml` (or `HEYLLAVE_DATABASE_URL`) and creates: the first tenant schema, the admin role, the admin user, and the membership that ties them together.

   ```bash
   go run ./cmd/seed \
     --tenant "Acme Auto Repair" \
     --email admin@acme.example \
     --password "SuperSecret123!" \
     --name "Alice"
   ```

   Or run it interactively and answer the prompts:

   ```bash
   go run ./cmd/seed
   ```

   Expected output:

   ```
   Tenant created: Acme Auto Repair (tenant_acme)
   Admin user:    admin@acme.example
   Membership:    role=admin, status=active
   ```

   Under the hood, the seed binary executes the same flow as `POST /api/v1/tenants`:

   1. Insert a row into `public.tenants`.
   2. `CREATE SCHEMA tenant_acme`.
   3. Apply every file in `migrations/core/tenant/` to `tenant_acme`.
   4. Apply `migrations/modules/<name>.sql` for every module listed under `tenant.default_modules` in `heyllave.yaml`.
   5. Seed the default roles.
   6. Insert one row per provisioned module into `public.tenant_modules`.
   7. Create the admin user and membership.

   Everything runs inside a single transaction — if any step fails, the schema is dropped and the database is left in its previous state.

3. ## Log In

   Exchange credentials for an access token. The server resolves the tenant from the user's membership.

   ```bash
   curl -s -X POST http://localhost:8080/api/v1/auth/login \
     -H "Content-Type: application/json" \
     -d '{
       "email": "admin@acme.example",
       "password": "SuperSecret123!",
       "subdomain": "acme"
     }' | jq
   ```

   Response `200 OK`:

   ```json
   {
     "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
     "refresh_token": "dGhpcyBpcyBhIHJlZnJlc2ggdG9rZW4...",
     "expires_in": 900
   }
   ```

   Export the access token for the remaining steps:

   ```bash
   export TOKEN="eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
   ```

4. ## Check Your Identity

   Confirm the token is valid and inspect the resolved claims: user, tenant, and active permissions.

   ```bash
   curl -s http://localhost:8080/api/v1/auth/me \
     -H "Authorization: Bearer $TOKEN" | jq
   ```

   Response `200 OK`:

   ```json
   {
     "user": {
       "id": "usr_01J9...",
       "email": "admin@acme.example",
       "name": "Alice"
     },
     "tenant": {
       "id": "ten_02K1...",
       "name": "Acme Auto Repair",
       "subdomain": "acme",
       "schema_name": "tenant_acme"
     },
     "permissions": [
       "users:read",
       "users:write",
       "audit:read"
     ]
   }
   ```

5. ## Verify the Tenant Schema

   The CLI can confirm that the tenant's schema was created and that its core + module tables are present.

   ```bash
   heyllave db status
   ```

   ```
   Container:   heyllave-local-db  [running]
   Database:    platform

   Public schema:
     migrations: applied (9 files)

   Tenant schemas:
     tenant_acme
       core/tenant:  applied (10 files)
       modules:      users, audit
   ```

   You can also connect directly to verify with `psql`:

   ```bash
   psql "postgres://platform:platform@localhost:5432/platform" \
     -c "\dn" | grep tenant
   ```

   ```
    tenant_acme | platform
   ```

</Steps>

## What Happened Under the Hood

When you ran the bootstrap endpoint, the platform diffed the embedded `migrations/core/public/` files against the live `public` schema and applied any DDL needed to bring it in line.

When you ran `cmd/seed`, the platform:

1. Inserted a row in `public.tenants` for the new tenant.
2. Executed `CREATE SCHEMA tenant_acme`.
3. Set `search_path` to `tenant_acme, public` for the transaction.
4. Applied every file in `migrations/core/tenant/` to the new schema (sessions, audit_logs, events_outbox, hooks, notifications, storage, settings, search, approvals, state snapshots, translations).
5. Applied `migrations/modules/<name>.sql` for each module in `tenant.default_modules`.
6. Inserted the admin role, user, and membership.

From this point on, every authenticated request for this tenant runs inside a transaction with `SET LOCAL search_path = tenant_acme, public`. Unqualified table names route to `tenant_acme.*`; any reference to `public.users`, `public.tenants`, etc. falls through automatically.

## Adding More Tenants

Once the platform is bootstrapped, additional tenants are created via the HTTP API rather than the seed binary:

```bash
curl -s -X POST http://localhost:8080/api/v1/tenants \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Globex",
    "subdomain": "globex"
  }' | jq
```

The endpoint runs the same provisioning flow as the seed binary, minus the interactive admin-creation step.
