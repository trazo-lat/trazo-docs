## Step 1 — Start the Local Database

The `heyllave db local` command starts a PostgreSQL 15 container named `heyllave-local-db` using Docker. It creates the platform database and the default platform user automatically.

```bash
heyllave db local
```

Expected output:

```
Starting heyllave-local-db...
Container heyllave-local-db started on port 5432
Platform database ready.
```

<Aside type="note">
  The container is configured to match the default `heyllave.yaml` connection string:
  `postgres://platform:platform@localhost:5432/platform?sslmode=disable`

  If you change the port or credentials in your config, pass the corresponding flags:

  ```bash
  heyllave db local --port 5433 --password mysecret
  ```
</Aside>

### Check Database Status

```bash
heyllave db status
```

```
Container:  heyllave-local-db  [running]
Host:       localhost:5432
Database:   platform
User:       platform
```

### Verify Connectivity

```bash
heyllave db ping
```

```
OK  connected to postgres://platform:****@localhost:5432/platform
```

If `ping` fails, ensure Docker is running and the container started without errors (`docker ps | grep heyllave-local-db`).

## Step 2 — Start the Server

The server boots without the public schema being populated. It will refuse most API calls until you bootstrap the database in step 3, but it must be running first so the bootstrap endpoint is reachable.



```bash
heyllave-server
```

The server starts and listens on `0.0.0.0:8080` by default. You should see structured log output similar to:

```json
{"level":"info","ts":"2026-03-03T10:00:00Z","msg":"Heyllave server starting","env":"local","port":8080}
{"level":"info","ts":"2026-03-03T10:00:01Z","msg":"Module registered","module":"users"}
{"level":"info","ts":"2026-03-03T10:00:01Z","msg":"Module registered","module":"audit"}
{"level":"info","ts":"2026-03-03T10:00:01Z","msg":"Event worker pool started","workers":10}
{"level":"info","ts":"2026-03-03T10:00:01Z","msg":"Server ready","addr":"0.0.0.0:8080"}
```

Override the port or log level without editing `heyllave.yaml` using flags or environment variables:

<Tabs>
  <TabItem label="Flags">
    ```bash
    heyllave-server --port 9090 --log-level debug
    ```
  </TabItem>
  <TabItem label="Environment Variables">
    ```bash
    HEYLLAVE_SERVER_PORT=9090 HEYLLAVE_LOG_LEVEL=debug heyllave-server
    ```
  </TabItem>
</Tabs>

## Step 3 — Bootstrap the Public Schema

With the server running, call the bootstrap endpoint to apply `migrations/core/public/` to the live database. This creates the `tenants`, `users`, `roles`, `permission_definitions`, etc. tables in the `public` schema. It is idempotent — running it again on an up-to-date database is a no-op.

```bash
curl -s -X POST http://localhost:8080/api/v1/admin/migrations/bootstrap | jq
```

The endpoint touches the `public` schema only. Per-tenant schemas are created later, when you seed the first tenant (see [Your First Tenant](/getting-started/first-tenant/)).

---

## Step 4 — Verify the Server is Running

```bash
curl http://localhost:8080/healthz
```

```json
{"status":"ok","version":"0.x.x","env":"local"}
```

A `200 OK` response confirms the server is up and connected to the database.

## Stopping the Local Database

When you are done developing, stop the container:

```bash
heyllave db local --stop
```

```
Container heyllave-local-db stopped.
```

The container and its data volume are preserved. The next time you run `heyllave db local` it will restart in seconds without re-running migrations.

To destroy the container and its data entirely (useful for a clean slate):

```bash
heyllave db local --destroy
```

<Aside type="caution">
  `--destroy` permanently deletes all local database data. You will need to re-bootstrap the public schema (and re-seed any tenants) afterwards.
</Aside>

## Typical Development Loop

```bash
# Start everything
heyllave db local
heyllave-server &

# Once, after a fresh database, bootstrap the public schema
curl -X POST http://localhost:8080/api/v1/admin/migrations/bootstrap

# Develop, make changes...

# Stop when done
kill %1
heyllave db local --stop
```

Continue to [Your First Tenant](/getting-started/first-tenant/) to seed the first tenant and admin user and make your first authenticated API request.
