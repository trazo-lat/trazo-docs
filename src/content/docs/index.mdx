## Why Trazo?

<CardGrid>
  <Card title="Schema-per-Tenant" icon="seti:db">
    Every tenant gets its own PostgreSQL schema — complete data isolation with no row-level filtering leaks. Back up, restore, or migrate tenants independently.
  </Card>
  <Card title="Pluggable Modules" icon="puzzle">
    Foundation, domain, and custom module types. Each module owns its routes, services, repositories, migrations, and event subscriptions. Add business logic without touching the core.
  </Card>
  <Card title="Full Audit Trail" icon="document">
    Every state change is recorded with RFC 6902 JSON Patch diffs. Query history by resource, actor, action, or time range. Mask sensitive fields automatically.
  </Card>
  <Card title="Event-Driven" icon="rocket">
    Transactional outbox pattern guarantees event delivery. Subscribe to domain events synchronously or asynchronously. Dead letter queue for failed processing.
  </Card>
  <Card title="Permission Scoping" icon="approve-check-circle">
    Fine-grained permissions with SQL-level scope enforcement. Built-in scopes: all, own, assigned, team, created. Scopes translate directly to WHERE clauses.
  </Card>
  <Card title="CLI Tooling" icon="terminal">
    `trazo` CLI handles local development, database management, migrations, and module validation. One command to spin up a local PostgreSQL instance.
  </Card>
</CardGrid>

## Two-Repo Architecture

Trazo is split into two Go repositories:

| Repository | Purpose | Key Tech |
|------------|---------|----------|
| `trazo-core-service` | API server, Core struct, domain modules | Go 1.24, Huma v2, chi, sqlx, squirrel |
| `trazo-cli` | Database management, migrations, module tooling | Go 1.24, Cobra, Docker SDK |

Both share the same `trazo.yaml` configuration format and connect to the same PostgreSQL instance.
