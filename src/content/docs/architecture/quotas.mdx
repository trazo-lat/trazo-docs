The quota engine gates and tracks per-tenant feature usage. Every tenant is implicitly subject to the engine — there is no enable/disable. Modules register metrics during bootstrap and call `Check`, `Reserve`, `Release`, `Record`, or `IsEnabled` at the boundaries where a quota should apply.

The engine is built around three primitives:

| Primitive | Storage | Use case |
|---|---|---|
| **Metric** | in-memory registry | The name + kind + unit of a thing being measured (e.g. `tenant.users.count`, `feature.whatsapp_channel`). |
| **Plan / plan_quota** | `public.plans`, `public.plan_quotas` | Per-tier limits assigned to tenants by `tenant_plans`. |
| **Quota rule** | `public.quota_rules` | Conditional per-tenant limit overrides gated by an expression. First match by priority wins. |

## Metric kinds

| Kind | Storage shape | Semantics |
|---|---|---|
| **counter** | current absolute value | Reserve increments, Release decrements, limit caps the absolute total. Example: number of seats. |
| **meter** | current value within a rolling window (day / month / year) | Reserve and Record both add to the window's current value; the engine rolls the window forward when expired. Example: API calls per month. |
| **flag** | boolean (`limit_value` 0 = off, non-zero = on) | `IsEnabled` reads plan + rules; no usage tracking. Example: `feature.whatsapp_channel`. |

`limit_value = -1` is the sentinel for "unlimited" on counters and meters. Reserve always succeeds for unlimited limits.

## Resolution chain

When `Reserve(ctx, metric_key, amount)` runs, the engine resolves the effective limit in priority order. The first level that produces a value wins:

```
1. quota_rules   — first enabled rule (priority DESC) whose condition matches the request context
2. tenant_quota_overrides — non-expired override row for (tenant, metric)
3. plan_quotas   — limit attached to the tenant's effective plan
4. unlimited     — engine fails open when no information exists (unknown metric or unconfigured tenant)
```

Unknown metric keys always fail open: a typo in a caller cannot block writes. Compile errors in a rule's condition are logged and the rule is skipped for that request, so a single bad rule never blocks `Reserve` for the whole tenant.

## Declaring metrics in `module.yaml`

Metrics are declared declaratively, alongside permissions and events. At bootstrap, `quotas.RegisterFromModules` parses every module's `module.yaml` and registers the metrics on the engine. The `Module` field on each metric is forced from the registry — YAML cannot spoof another module's namespace.

```yaml
# internal/modules/domain/conversations/module.yaml
name: conversations

quotas:
  - key: feature.whatsapp_channel
    name: WhatsApp channel
    kind: flag
    description: Enables the WhatsApp Business adapter for outbound and inbound messages.

  - key: feature.web_channel
    name: Web chat channel
    kind: flag

  - key: conversations.messages
    name: Messages
    kind: meter
    default_reset: month
    unit: messages
```

Fields:

| Field | Required | Type | Description |
|---|---|---|---|
| `key` | yes | string | Globally unique identifier. By convention `<module>.<metric>` or `feature.<flag>`. |
| `name` | yes | string | Human-readable display name. |
| `kind` | yes | `counter` \| `meter` \| `flag` | Storage semantics. |
| `default_reset` | meters only | `day` \| `month` \| `year` | Rolling-window length for meters. |
| `unit` | no | string | Display unit (e.g. `users`, `messages`, `bytes`). |
| `description` | no | string | Tooltip / API documentation copy. |

Cross-cutting platform metrics (`tenant.users.count`, `tenant.modules.enabled`, `tenant.storage.bytes`, `tenant.api_calls`) are still registered at the platform level — modules only declare metrics they own.

## Calling the engine

Modules access the engine through `core.Quotas`:

```go
// Counter: reserve before insert, release on rollback
if s.core.Quotas != nil {
    if _, qErr := s.core.Quotas.Reserve(ctx, quotas.MetricUsersCount, 1); qErr != nil {
        return nil, qErr // → 402 Payment Required, CORE.QUOTA.EXCEEDED
    }
}

if err := s.repo.Create(ctx, m); err != nil {
    if s.core.Quotas != nil {
        _ = s.core.Quotas.Release(ctx, quotas.MetricUsersCount, 1)
    }
    return nil, fmt.Errorf("creating membership: %w", err)
}

// Flag: branch on feature availability
if on, _ := s.core.Quotas.IsEnabled(ctx, "feature.whatsapp_channel"); on {
    // enable WhatsApp adapter
}

// Meter: record usage after a successful send
_ = s.core.Quotas.Record(ctx, "conversations.messages", 1)
```

`Reserve` is atomic: limit check and increment happen inside a single transaction. On limit-exceeded it returns a `*core.Error` with HTTP 402 and code `CORE.QUOTA.EXCEEDED`. The error params include `metric`, `current`, `limit`, `requested`, and `plan` for i18n templating.

## Plans and tenant assignment

Plans live in `public.plans` (the global catalog). Each plan attaches limits per metric in `public.plan_quotas`. Tenants are assigned to a plan via `public.tenant_plans`; tenants without an assignment fall back to the configured `quotas.default_plan` slug (default `"free"`).

| Table | Purpose |
|---|---|
| `public.plans` | Catalog of plan tiers (free, pro, enterprise...). |
| `public.plan_quotas` | Per-plan limit per metric. PK `(plan_id, metric_key)`. |
| `public.tenant_plans` | Which plan each tenant is on. PK `(tenant_id)`. |
| `public.tenant_quota_overrides` | Per-tenant single-value override for a metric. PK `(tenant_id, metric_key)`. |
| `public.quota_rules` | Per-tenant **conditional** overrides. Multiple rows per `(tenant_id, metric_key)`. |
| `tenant_<slug>.quota_usage` | Current usage counters and meter windows. Lives in the tenant schema. |

The default plan catalog (`free`, `pro`, `enterprise`) is seeded at bootstrap by `quotas.SeedDefaults` and is idempotent — re-running upserts the same plan definitions without dropping tenant assignments.

## Conditional rules

Quota rules are the headline feature that lets an admin **split** a plan-level quota across runtime conditions without forking the plan. Each rule attaches a priority and a [heyllave/query](https://github.com/heyllave/query) predicate. When `Reserve` runs, the engine evaluates rules `priority DESC` and uses the first one whose condition is true.

Conditions use the same URL-safe filter syntax as the platform's search bars (`state=draft AND total>50000`). No quotes around enum-like values, no `==`, no `&&` — predicates that read naturally and round-trip through a query string.

### Use cases

- **Geographic caps**: EU tenants on `pro` get 5,000 messages/month; the global cap is 10,000.
- **Role-based limits**: admins get the full limit, members get a fraction.
- **Trial bumps**: tenants flagged as trial get a temporary higher cap that expires after a timestamp.
- **Off-hours throttling**: API calls limited differently during weekends or off-hours.
- **Tier overlays**: `enterprise` skips a hard cap that `pro` enforces, without touching either plan row.

### Expression environment

Every rule predicate sees the same env, populated on every `Reserve` / `Check`. Fields are reached via dotted paths (`tenant.metadata.region`, `now.weekday`).

```javascript
{
  "tenant":     { id, slug, status, name, metadata },
  "user":       { id, email, locale, timezone, metadata }, // empty outside request scope
  "membership": { id, role, permissions, metadata },        // empty outside request scope
  "plan":       { slug, name, status, metadata },           // resolved effective plan
  "now":        { timestamp, iso, hour, weekday, month, day, year }
}
```

Compile errors are surfaced at write time so the admin sees them up front. Missing fields at runtime degrade to "no match" so a rule referencing `membership.role` is still safe to write — it just won't match for service-principal calls without a membership in context.

### Example: split a plan limit by region

```http
POST /api/v1/tenants/tnt_acme/billing/rules
Content-Type: application/json

{
  "metric_key": "conversations.messages",
  "priority": 100,
  "condition": "tenant.metadata.region=EU",
  "limit_value": 5000,
  "reset_period": "month",
  "label": "EU regional cap"
}
```

A subsequent `Reserve` for an EU tenant returns the 5,000 limit; the resulting `QuotaDecision` carries `source: "rule"` and the `rule_id` so operators can trace the decision back to the rule.

### Example: stacked rules

Trial tenants get a one-time 500/day bump for the first 7 days; everyone else falls back to a 100/day baseline.

```http
# priority 100 — trial bump (matches first when the condition holds)
POST /api/v1/tenants/tnt_acme/billing/rules
{
  "metric_key": "conversations.messages",
  "priority": 100,
  "condition": "tenant.metadata.trial=true AND now.timestamp<tenant.metadata.trial_ends_at",
  "limit_value": 500,
  "reset_period": "day",
  "label": "Trial bump"
}

# priority 50 — free-tier fallback (no condition = always matches)
POST /api/v1/tenants/tnt_acme/billing/rules
{
  "metric_key": "conversations.messages",
  "priority": 50,
  "limit_value": 100,
  "reset_period": "day",
  "label": "Free baseline"
}
```

### Common expression patterns

```
plan.slug=enterprise                                      # tier gating
membership.role=admin                                     # role-based caps
tenant.metadata.region IN (EU, UK)                        # geo split (IN desugars to OR chain)
now.weekday>=1 AND now.weekday<=5 AND now.hour>=9         # business-hours only
tenant.status=trial                                       # lifecycle gating
NOT tenant.metadata.region=US                             # negation
```

<Aside type="tip">
An empty `condition` means "always match". Place a no-condition rule at a low priority as the catch-all fallback behind higher-priority gated rules.
</Aside>

## Decision provenance

Every `QuotaDecision` (returned by `Check` / `Reserve`) and every `QuotaSnapshot` (returned by `/billing/usage`) carries enough provenance for an operator to trace any limit back to its origin:

```json
{
  "metric_key": "conversations.messages",
  "allowed": true,
  "limit": 5000,
  "current": 1247,
  "remaining": 3753,
  "requested": 1,
  "reset_period": "month",
  "period_start": "2026-05-01T00:00:00Z",
  "period_end":   "2026-06-01T00:00:00Z",
  "source": "rule",
  "plan_slug": "pro",
  "rule_id": "qrule_conversations-messages-a4f81234",
  "rule_label": "EU regional cap"
}
```

`source` is one of:

| Value | Meaning |
|---|---|
| `rule` | A conditional rule matched. `rule_id` and `rule_label` are populated. |
| `override` | A `tenant_quota_overrides` row supplied the limit. |
| `plan` | The plan's `plan_quotas` row supplied the limit. `plan_slug` is populated. |
| `unregistered` | No metric or no information — engine failed open with `limit: -1`. |

## Admin API

All endpoints are namespaced under `/api/v1/billing/...` so the quota surface stays beside future invoice / subscription endpoints. Permissions are namespaced under `billing.*`.

| Method | Path | Permission | What it does |
|---|---|---|---|
| `GET`    | `/billing/plans` | `billing.plans.read` | List plan catalog. |
| `POST`   | `/billing/plans` | `billing.plans.write` | Create a plan. |
| `GET`    | `/billing/plans/{slug}` | `billing.plans.read` | Get a plan + its quotas. |
| `PUT`    | `/billing/plans/{slug}` | `billing.plans.write` | Replace a plan (full body). |
| `POST`   | `/billing/plans/{slug}/quotas` | `billing.plans.write` | Upsert a `(metric, limit)` line on a plan. |
| `DELETE` | `/billing/plans/{slug}/quotas/{metric_key}` | `billing.plans.write` | Remove a plan quota line. |
| `GET`    | `/tenants/{tenant_id}/billing/plan` | `billing.plans.read` | Get the tenant's effective plan. |
| `PUT`    | `/tenants/{tenant_id}/billing/plan` | `billing.assign` | Assign a plan to a tenant. |
| `GET`    | `/tenants/{tenant_id}/billing/usage` | `billing.usage.read` | Snapshot of usage + effective limits. |
| `GET`    | `/tenants/{tenant_id}/billing/overrides` | `billing.assign` | List active overrides. |
| `PUT`    | `/tenants/{tenant_id}/billing/overrides/{metric_key}` | `billing.assign` | Upsert an override. |
| `DELETE` | `/tenants/{tenant_id}/billing/overrides/{metric_key}` | `billing.assign` | Remove an override. |
| `GET`    | `/tenants/{tenant_id}/billing/rules` | `billing.rules.write` | List conditional rules. |
| `POST`   | `/tenants/{tenant_id}/billing/rules` | `billing.rules.write` | Create a conditional rule. |
| `PATCH`  | `/tenants/{tenant_id}/billing/rules/{rule_id}` | `billing.rules.write` | Partially update a rule. |
| `DELETE` | `/tenants/{tenant_id}/billing/rules/{rule_id}` | `billing.rules.write` | Remove a rule. |

Rule conditions are compiled before persistence, so syntactically invalid expressions return `400 Bad Request` rather than poisoning runtime resolution.

## Database schema

```sql
-- public.quota_rules — conditional per-tenant quota limits
CREATE TABLE quota_rules (
    id           UUID PRIMARY KEY,
    public_id    TEXT NOT NULL UNIQUE,
    tenant_id    UUID NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
    metric_key   TEXT NOT NULL,
    priority     INT NOT NULL DEFAULT 0,
    condition    TEXT NOT NULL DEFAULT '',     -- heyllave/query predicate; empty = always match
    limit_value  BIGINT NOT NULL,              -- -1 = unlimited, 0 = hard zero
    reset_period TEXT NOT NULL DEFAULT 'none', -- none | day | month | year
    label        TEXT NOT NULL DEFAULT '',
    enabled      BOOLEAN NOT NULL DEFAULT TRUE,
    metadata     JSONB NOT NULL DEFAULT '{}',
    created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_quota_rules_tenant_metric
    ON quota_rules(tenant_id, metric_key, priority DESC)
    WHERE enabled = TRUE;
```

`tenant_<slug>.quota_usage` (per-tenant schema) holds the live counters and meter windows. The schema-aware DB layer routes reads and writes to the right tenant schema automatically once the tenant is in context.

## Configuration

```yaml
# heyllave.yaml
quotas:
  default_plan: "free"      # slug applied to tenants with no tenant_plans row
  plan_cache_ttl: 60s       # in-memory cache for resolved plan bundles
```

`InvalidatePlanCache` is called automatically when a plan, plan quota, override, or tenant assignment changes via the admin API, so cached limits are never stale for more than a handler hop.

## Operational notes

- **Failure-open by design**: unregistered metrics and missing plan rows resolve to `unlimited`. A typo in a caller never blocks writes.
- **One rule cannot block the tenant**: a runtime error in a single rule's expression is logged and the rule is skipped; the next rule in priority order still runs.
- **Idempotent registration**: re-registering the same metric key is a no-op; conflicting kinds for the same key log a warning and keep the first registration. This makes hot-reload and module re-init safe.
- **Counter / meter writes need a tenant in context**. Flag reads work without one (they read plan + rules only). Calling `Reserve` for a counter without a tenant returns `CORE.QUOTA.NO_TENANT`.
