title: Policies (Write-Time Entity Invariants)
description: Cross-entity constraints checked at create / update / delete. Distinct from quotas (gate usage) and permissions (gate actors) — policies assert what the entity itself must look like.
---

import { Aside } from '@astrojs/starlight/components';

The **policy engine** gates entity writes by evaluating predicates against a caller-supplied env. It's the third leg of the access model:

| Surface | Gates | Example |
|---|---|---|
| Permission | the actor | "this user has `repairs.update.assigned`" |
| Quota | usage volume | "this tenant has fewer than 25 users" |
| **Policy** | the entity | "memberships with role=admin must include MFA in auth_methods" |

A module calls `core.Policies.Check` before persisting an entity. If at least one matching policy is `deny`-class, the engine returns a `*PolicyResult` with `Denied=true` and the caller translates that into a `*core.Error`. `warn`-class policies surface as advisory violations alongside.

## Anatomy of a policy

```json
{
  "id": "pol_membership-admin-mfa-a4f81234",
  "tenant_id": null,
  "entity_type": "membership",
  "action": "create",
  "condition": "entity.role=role_admin",
  "on_violate": "deny",
  "violation_code": "MEMBERS.POLICY.ADMIN_REQUIRES_MFA",
  "message": "Admin memberships require an MFA auth method",
  "priority": 100,
  "enabled": true
}
```

| Field | Description |
|---|---|
| `tenant_id` | `null` = global policy applied to every tenant; otherwise scoped to that tenant |
| `entity_type` | The entity the policy gates — `membership`, `invoice`, `tenant`, etc. |
| `action` | `create` / `update` / `delete` / `*` (all). Wildcards apply broadly. |
| `condition` | A [heyllave/query](https://github.com/heyllave/query) predicate that describes the **forbidden** state |
| `on_violate` | `deny` (block with error) or `warn` (return as advisory only) |
| `violation_code` | Error code returned when a deny policy matches |
| `message` | Human-readable message — surfaces in API errors and admin UI |
| `priority` | Higher evaluated first; doesn't change correctness, just collection order |

<Aside type="tip">
**The predicate matches the forbidden state.** Policy authors often want to write "X must have Y" but the engine's condition is "X without Y" — when the predicate is true, the policy is violated. This keeps the language consistent with quota rules and search filters.
</Aside>

## Calling Check

```go
// Inside the members service's Create flow
env := map[string]any{
    "entity": map[string]any{
        "role":         input.Role,
        "auth_methods": input.AuthMethods,
    },
    "tenant":     buildTenantEnv(ctx),
    "membership": buildMembershipEnv(ctx),
    "plan":       buildPlanEnv(ctx),
}

result, err := s.core.Policies.Check(ctx, "membership", "create", env)
if err != nil {
    return nil, err
}
if result.Denied {
    return nil, asViolationError(s.core, result)
}
// surface advisory warnings to the caller (in a header, response field, etc.)
for _, v := range result.Violations {
    if v.OnViolate == core.PolicyActionWarn {
        s.core.Log.Logger(ctx).WithModule("members").Warn(
            "policy advisory",
            map[string]any{"code": v.Code, "message": v.Message},
        )
    }
}
```

`asViolationError` is your module's helper for turning a `PolicyResult` into a `*core.Error` — usually by picking the highest-priority deny violation and mapping its code to a `core.ErrorCode`.

## Predicate language

The env is whatever the caller assembles, but the engine's per-path typer recognises five well-known roots:

| Root | Typical contents |
|---|---|
| `entity` | The entity being written (role, auth_methods, name, status, metadata fields, etc.) |
| `tenant` | The tenant in context — `id`, `slug`, `status`, `metadata.*` |
| `user` | The acting user — `id`, `email`, `locale`, `timezone`, `metadata.*` |
| `membership` | The acting membership — `role`, `permissions`, `metadata.*` |
| `plan` | The tenant's effective plan — `slug`, `name`, `metadata.*` |
| `now` | Wall clock — `timestamp` (datetime), `weekday`/`hour`/`month`/`day`/`year` (int) |

Dotted fields ending in `_at` or `_date` under `entity`/`tenant`/`user`/`membership`/`plan` are typed as `datetime` so date comparisons work without manual casts. Everything else under those roots is `text` for safe string equality.

## Common patterns

### Admin role requires MFA

```
entity.role=role_admin
```

Combined with a join: the caller writes `entity.auth_methods` as the joined comma-separated string `password,mfa,webauthn`. The condition becomes `entity.role=role_admin AND NOT entity.auth_methods=*mfa*` using wildcards. Or — cleaner — the caller flattens `mfa_enrolled: true|false` into the env and writes:

```
entity.role=role_admin AND entity.mfa_enrolled=false
```

### Enterprise plan requires billing email

```
plan.slug=enterprise AND entity.billing_email=""
```

Forbids creating a tenant on `enterprise` without a billing email on the tenant record.

### Invoice cannot issue without lines

```
entity.line_count=0
```

Caller passes `entity.line_count` in the env at issue time. Predicate matches when no lines exist → deny.

### Region-locked tenant must have a region

```
plan.metadata.region_locked=true AND entity.region=""
```

### Trial accounts can't issue invoices > $1000

```
plan.slug=trial AND entity.total_cents>100000
```

A meaningful numeric comparison — the per-path typer infers `entity.total_cents` is integer because the RHS literal is an integer.

## Tenant scoping

Policies are either **global** (`tenant_id IS NULL` — apply to every tenant) or **tenant-scoped** (`tenant_id` set — apply only to that tenant). The engine's `Check` returns the union: every global policy plus every policy scoped to the tenant in `ctx`.

Tenant-scoped policies are useful when a single tenant wants stricter rules than the platform defaults — e.g. an enterprise tenant mandating every new membership comes with a Slack handle in `entity.metadata.slack`.

## Caching

Policies are cached in memory for the configured TTL (default 60s) keyed by `(entity_type, action)`. Admin tooling should call `Engine.InvalidateCache()` after any policy mutation; the HTTP CRUD layer does this automatically once it lands.

A failure to compile a policy's condition is logged and the policy is **skipped** for that request, so a single bad expression cannot block every write for the tenant.

## Designing a new policy

1. **Pick the entity type and action.** Be specific — `membership/create` is better than `membership/*` because the env shape changes between create and update.
2. **Decide deny vs warn.** Deny when the rule is invariant (admin must have MFA, period). Warn when it's a guardrail (we'd like billing email but it's not mandatory yet).
3. **Write the predicate as the forbidden state.** "Y must contain X" → "Y does not contain X".
4. **Set a code that callers can pattern-match.** `MEMBERS.POLICY.ADMIN_REQUIRES_MFA` reads at the call site; raw English in the message is for human consumption.
5. **Test the env shape.** The caller's env must satisfy the predicate's field references — a typo silently degrades to "no match" (the rule never fires).
